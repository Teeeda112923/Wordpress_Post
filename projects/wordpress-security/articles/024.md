# WordPressの脆弱性｜確認方法と対処手順

WordPressの脆弱性の9割は、本体ではなくプラグインにあります。2025年に見つかった脆弱性のうち、コアに起因するものはごく少数でした。つまり対策の中心は、本体の更新ではなく、導入しているプラグインとテーマの棚卸しと照合です。この記事では、脆弱性が存在する3つの層を整理し、自分のサイトが影響を受けるかを確認する手順と、更新・停止・代替の判断基準を示します。

## WordPressの脆弱性とは何か

WordPressの脆弱性とは、本体・テーマ・プラグインのプログラムに存在する欠陥のうち、悪用されると改ざん、情報の窃取、管理権限の乗っ取りにつながるもののことです。設定ミスとは区別されます。設定ミスは運営者が直せますが、脆弱性はプログラム自体の欠陥であり、開発者が修正版を出すまで根本的には解消できません。

脆弱性は、存在する場所によって3つの層に分かれます。層ごとに影響範囲も対処方法も違うため、まずこの区別を押さえてください。

| 層 | 影響を受ける範囲 | 主な対処 |
|---|---|---|
| コア（WordPress本体） | すべてのWordPressサイト | 本体の更新 |
| プラグイン | そのプラグインを使っているサイト | 更新、停止・削除、代替への移行 |
| テーマ | そのテーマを使っているサイト | 更新、テーマの変更 |

影響範囲だけを見ると、コアの脆弱性が最も深刻に思えます。しかし実際の発生件数は逆で、リスクは圧倒的にプラグイン側に偏っています。この事実を知らないまま「本体を最新にしているから安全」と考えるのが、最も危険な状態です。

## 数字で見る、脆弱性がどこにあるのか

脆弱性の分布は、感覚ではなく公表されているデータで確認できます。Patchstackが2026年2月に公表した年次レポートには、WordPressエコシステム全体を対象とした集計が掲載されています。

対策の優先順位を決めるうえで押さえておくべき数値は、次のとおりです。これらはすべて2025年の実績です。

| 指標 | 数値 |
|---|---|
| 新たに見つかった脆弱性 | 11,334件（前年比42%増） |
| プラグインに起因する割合 | 91% |
| テーマに起因する割合 | 9% |
| WordPressコアで報告された件数 | 6件（いずれも優先度は低いと報告） |
| 公開時点で修正版が存在しなかった割合 | 46% |
| 公開から大規模な悪用までの中央値 | 5時間 |

この表から読み取れることは3つです。まず、WordPress本体は堅牢です。これだけ広く使われているソフトウェアで年間6件という数字は、極めて少ない部類に入ります。

次に、リスクはプラグインに集中しています。対策としてまず取り組むべきは、本体の更新ではなく、プラグインの数を減らし、使っているものの状態を把握することです。

そして3つ目が最も重要です。公開された脆弱性のおよそ半数は、修正版が存在しない状態で公表されています。さらに、公開から悪用が始まるまでの中央値は5時間です。月に一度まとめて更新する運用では、この速度に追いつけません。

## なぜプラグインに集中するのか

プラグインに脆弱性が偏るのは、開発者の技術力が低いからではありません。誰でも拡張機能を公開できるという、WordPressの仕組みそのものから生じる結果です。

原因は主に3つあります。いずれも、オープンな仕組みの利点と表裏一体の関係にあります。

- **誰でも公開できる**：公式ディレクトリには数万のプラグインが登録されており、その多くは個人が無償で開発しています。セキュリティの専門知識を持たない開発者も参加できることが、多様性の源であり、同時にリスクの源でもあります
- **開発が止まっても公開され続ける**：開発者が離れても、プラグイン自体は残ります。脆弱性が見つかっても修正されず、それを使い続けているサイトが放置されます
- **修正の義務がない**：無償で提供している以上、開発者に修正の責任はありません。前述のレポートでも、報告された脆弱性の半数近くが公開までに修正されなかったと指摘されています

したがって、プラグインを選ぶときに見るべきなのは機能ではなく、**維持されているかどうか**です。最終更新日が1年以上前のものは、脆弱性が見つかっても修正されない可能性が高いと考えてください。

## 脆弱性の種類と、悪用されたときに起きること

脆弱性は種類によって被害の内容が変わります。名前だけを覚える必要はありませんが、どの種類が自分にとって危険かを判断できると、対応の優先順位が決めやすくなります。

WordPressのプラグインで報告されることが多い脆弱性は、次の4種類です。上にあるものほど、被害が直接的で深刻になります。

| 種類 | 悪用されると何が起きるか | 危険度の目安 |
|---|---|---|
| 任意のファイルアップロード・コード実行 | サーバー上に不正なプログラムを設置され、サイト全体を乗っ取られる | 最も高い |
| 権限昇格・アクセス制御の不備 | 一般ユーザーや未ログインの攻撃者が、管理者と同じ操作を行える | 高い |
| SQLインジェクション | データベースを直接操作され、会員情報の窃取や管理者の追加が行われる | 高い |
| クロスサイトスクリプティング（XSS） | 訪問者のブラウザで不正なスクリプトが実行され、情報の窃取や別サイトへの誘導が起こる | 中程度 |

とくに注意したいのが、2行目のアクセス制御の不備です。前述のPatchstackのレポートでは、防御手段によって遮断された攻撃のうち相当な割合がこの種類だったと報告されています。

この種類が厄介なのは、攻撃が正常なアクセスと見分けにくい点です。不正な文字列を送り込むわけではなく、本来は許可されていない操作を、正規の手順で実行してしまうためです。外部の防御サービスでは検知しにくく、結局はプラグイン側を直すしかありません。

## テーマの脆弱性も見落とさない

脆弱性の議論はプラグインに集中しがちですが、テーマにも脆弱性は存在します。前述のレポートでは、2025年に見つかった脆弱性の9%がテーマに起因していました。件数はプラグインより少ないものの、無視できる水準ではありません。

テーマ特有の注意点は3つあります。プラグインとは事情が異なるため、分けて考えてください。

- **使っていないテーマも攻撃対象になる**：有効化していなくてもファイルは残ります。予備を1つ残し、それ以外は削除してください
- **正規の配布元以外から入手したテーマは危険**：有料テーマを無償で配布しているサイトがありますが、こうしたファイルには不正なコードが埋め込まれていることがあります。公式ディレクトリか、開発元から入手してください
- **カスタマイズした部分は更新で消える**：テーマを直接編集していると、更新のたびに変更が失われます。子テーマを使うのが原則です

3つ目は脆弱性そのものの話ではありませんが、実務上は重要です。更新すると表示が崩れるという理由でテーマの更新を止めているサイトは珍しくありません。その状態が、脆弱性を放置する原因になります。

## 自分のサイトが影響を受けるかを確認する

確認は3つの手順で進めます。導入しているものを洗い出し、脆弱性情報と照合し、影響を受けるバージョンかどうかを判定する、という流れです。

### 手順1：導入しているプラグインとテーマを棚卸しする

管理画面の「プラグイン」→「インストール済みプラグイン」を開き、名前とバージョンを控えます。停止中のものも対象です。有効化していなくてもファイルはサーバー上に残っており、攻撃者はそれを直接呼び出せる場合があります。

テーマも同様に、「外観」→「テーマ」から確認します。使っていないテーマが残っていれば、それも棚卸しの対象です。

### 手順2：脆弱性情報と照合する

控えた名前とバージョンを、脆弱性情報のデータベースで検索します。無料で使える主な情報源は、JVN iPedia（IPA・JPCERT/CCが運営、日本語）、WPScan Vulnerability Database（WordPress専門）、Patchstack Databaseです。

CVEとは、脆弱性ひとつひとつに割り当てられる識別番号のことです。同じ脆弱性が複数のサイトで報告されても、CVE番号が一致していれば同一のものだと判断できます。

各データベースの使い分けと、確認する順番については別記事で詳しく説明しています。

### 手順3：影響バージョンを確認する

脆弱性情報には、必ず「影響を受けるバージョン」が記載されています。ここが確認の要です。自分が使っているバージョンがその範囲に含まれていなければ、その脆弱性の影響は受けません。

見落としやすいのが、修正版のバージョン表記です。「1.5.2未満」と書かれている場合、1.5.2そのものは修正済みです。範囲の境界を読み違えると、対応済みのものを再度直そうとしたり、逆に影響を受けているのに見逃したりします。

## 見つかったときに何を選ぶか

対処の選択肢は、更新・停止・代替の3つです。修正版があるかどうか、そのプラグインが業務に必要かどうかで決まります。

判断は次のように整理できます。迷ったときは、停止して様子を見るのが最も安全です。動かないことによる不便は元に戻せますが、侵入された場合は元に戻せません。

| 状況 | 選択 | 理由 |
|---|---|---|
| 修正版が公開されている | 更新する | 最も確実。バックアップ後に適用する |
| 修正版がない／開発が停止している | 停止・削除する | 脆弱なコードは、停止すれば呼び出されなくなる |
| 機能が業務に必要で、止められない | 代替を探す | 同じ機能を持つ、維持されている製品へ移行する |
| 修正版はあるが、更新で不具合が出る | 検証環境で確認してから適用 | 影響範囲を確かめたうえで本番へ |

代替への移行は時間がかかります。脆弱性が公表されてから探し始めるのでは間に合いません。重要な機能を担っているプラグインについては、平時のうちに代替候補を把握しておくと、対応の速度が変わります。

## 緊急対応が必要かどうかを見分ける

すべての脆弱性に即日対応する必要はありません。緊急度は、悪用のしやすさと、実際に狙われているかどうかで決まります。

判断材料になるのは、次の3点です。脆弱性情報のページには、これらが記載されていることが多いため、確認してください。

- **認証が必要かどうか**：ログインしていない攻撃者でも実行できる脆弱性は、緊急度が高くなります。自動化された攻撃の対象になりやすいためです
- **すでに悪用が観測されているか**：実際の攻撃が確認されている脆弱性は、最優先で対応します
- **深刻度（CVSS）の値**：脆弱性の深刻さを数値で示した指標です。ただし数値が高くても、悪用条件が厳しければ緊急度は下がります

数値だけで判断しないでください。深刻度が中程度でも、認証不要で悪用でき、対象のプラグインを自分が使っているのであれば、そちらのほうが優先されます。

## プラグインを選ぶ段階でリスクを減らす

脆弱性への対応で最も効くのは、公表されてから動くことではなく、そもそも脆弱性を抱えにくいプラグインを選ぶことです。導入時の数分の確認が、その後の対応工数を決めます。

導入前に見るべき項目は4つです。機能が要件を満たしていても、この4つで不安があるなら、別の製品を探してください。

| 確認項目 | 判断の目安 |
|---|---|
| 最終更新日 | 1年以上更新がないものは、脆弱性が見つかっても修正されない可能性がある |
| 有効インストール数 | 利用者が多いほど、問題が早く発見・報告されやすい |
| サポートフォーラムの応答 | 質問に開発者が反応しているか。放置されていないか |
| 過去の脆弱性への対応 | 報告された脆弱性を、公開までに修正しているか |

とくに1つ目は、機械的に判断できる指標です。WordPress.orgのプラグインページには最終更新日が表示されており、長期間更新されていないものには警告が表示されることもあります。

そして、導入するかどうかの判断で最も重要な問いは「そのプラグインがなければ困るか」です。あれば便利という理由で入れたプラグインが、数年後に脆弱性の入口になる。この構図が、実際の侵害では繰り返されています。機能を1つ追加するたびに、守るべき対象が1つ増えると考えてください。

## 更新だけでは埋まらない空白がある

ここまでの手順を実行しても、防ぎきれない期間が残ります。前述のとおり、公開された脆弱性の46%は修正版が存在しない状態で公表されているためです。修正版がなければ、更新という手段は使えません。

この空白を埋める方法は、大きく3つです。どれも完全ではありませんが、組み合わせれば実質的なリスクは下がります。

- **プラグインの数を減らす**：使っていないものを削除すれば、影響を受ける確率がそのまま下がります。最も確実で、費用もかかりません
- **情報を早く知る**：公開から5時間で悪用が始まる以上、週次の確認では間に合いません。自分が使っているプラグインに関する情報だけが手元に届く仕組みを作ってください
- **代替手段を用意しておく**：修正版が出るまで停止できるかどうかを、平時に判断しておきます

なお、外部の防御サービスに期待しすぎないでください。前述のレポートでは、一般的な防御手段が実際の攻撃を防げた割合は限定的だったと報告されています。防御を入れているから確認しなくてよい、とはなりません。

## 自分のサイトの状態を確認する

脆弱性の照合を始めるには、まず「何をどのバージョンで使っているか」が分かっていなければなりません。ここが曖昧なままだと、照合の対象すら定まりません。

<!-- CTA -->
筆者が開発・公開しているCyberNote Security Checkerは、この現状把握のための無料の診断プラグインです。WordPress本体・PHP・プラグイン・テーマの更新状況（バージョン鮮度3項目）と、設定のハードニング7項目の計10項目を管理画面から確認できます。結果は要対応・改善推奨・問題なしの3段階で表示されます。

このプラグインは診断ツールであり、攻撃を遮断するものではありません。すべての脆弱性を検出できるわけでもないため、防御用のプラグインやバックアップと組み合わせて使ってください。導入は、管理画面の「プラグイン」→「新規プラグインを追加」で `CyberNote` と検索します。

## よくある質問

**Q. WordPressは脆弱性が多いのでしょうか。**
A. 本体の脆弱性は少なく、2025年に報告されたのは6件です。件数の大半はプラグインとテーマに集中しています。

**Q. 最新版にしていれば安全ですか。**
A. 安全とは言い切れません。公開された脆弱性の46%は、修正版が存在しない状態で公表されています。

**Q. 有料のプラグインなら安全ですか。**
A. そうとは限りません。有料であっても脆弱性は報告されており、価格と安全性は直結しません。

**Q. 脆弱性が見つかったプラグインは、すぐ削除すべきですか。**
A. 修正版があれば更新してください。ない場合は、停止して代替を探すのが安全です。業務で使っている場合は、影響を確認してから判断します。

## まとめ

WordPressの脆弱性は、本体ではなくプラグインとテーマに集中しています。2025年に見つかった11,334件のうち91%がプラグイン、9%がテーマで、コアはわずか6件でした。確認の手順は、導入しているものを棚卸しし、JVN・CVE・WPScanなどで照合し、影響バージョンを判定する、という3段階です。見つかった場合は、修正版があれば更新、なければ停止か代替への移行を選びます。ただし公開された脆弱性の半数近くは修正版がないまま公表されるため、更新だけに頼る運用には限界があります。まずは使っていないプラグインを削除し、対象を減らすところから始めてください。
