# WordPress防御機能の役割分担｜WAFとプラグイン

WordPressの防御は、WAF、セキュリティプラグイン、CDN、サーバー標準機能という複数の層で担われます。これらは守る範囲が一部重なり、一部は抜けています。同じ機能を重複させると誤検知や無駄が生じ、抜けがあればそこが侵入口になります。この記事では、各層が守る範囲を比較し、防御をどこに配置するかを整理します。重複と抜けを見直す点検表としても使えます。

## 防御は複数の層で担われる

WordPressの防御は、1つの手段だけで完結するものではありません。WAF、セキュリティプラグイン、CDN、サーバー標準機能という複数の層が、それぞれ異なる範囲を守っています。この全体像を理解することが、過不足のない防御につながります。

複数の層があることには、利点と課題の両方があります。利点は、1つの層が破られても、別の層が防ぐことです。課題は、層の間で機能が重複したり、逆に抜けが生じたりすることです。

まず、それぞれの層が何かを整理します。役割と、提供される形態が異なります。

| 層 | 何を担うか | 提供形態 |
|---|---|---|
| WAF | 攻撃パターンの遮断 | サーバー、プラグイン、CDN |
| セキュリティプラグイン | ログイン保護、スキャンなど | プラグイン |
| CDN | 配信の高速化＋一部の防御 | 外部サービス |
| サーバー標準機能 | 基盤の保護、一部のWAF | サーバー |

これらは、守る範囲が一部重なっています。たとえば、WAFはサーバー、プラグイン、CDNのいずれでも提供されることがあります。この重なりを理解しないと、同じ機能を二重に持ったり、どこにも配置しなかったりします。以降、各層の役割を整理します。

## WAFの役割

WAFは、Webアプリケーションを狙う攻撃のパターンを検知して遮断する仕組みです。SQLインジェクションやクロスサイトスクリプティングといった、典型的な攻撃を防ぎます。

WAFの特徴は、攻撃をアプリケーションに到達する前に遮断できることです。WordPressやプラグインに脆弱性があっても、その脆弱性を突く攻撃パターンをWAFが遮断できる場合があります。修正版が出ていない脆弱性に対しても、一定の防御が期待できます。

WAFは、複数の場所で提供されます。どこで提供されるかによって、特徴が異なります。

- **サーバーのWAF**：多くのレンタルサーバーが標準で提供します。基盤に近い位置で動作します
- **プラグインのWAF**：総合型のセキュリティプラグインが備えます。WordPressの中で動作します
- **CDNのWAF**：一部のCDNサービスが提供します。サイトに到達する前の段階で動作します

重要なのは、これらのWAFを重複させないことです。複数のWAFが同時に動作すると、二重の検査で処理が重くなり、誤検知も増えます。どこか1つでWAFを担うのが基本です。

どのWAFを使うかは、環境によります。サーバーがWAFを提供しているなら、それを使うのが手軽です。その場合、プラグインのWAFは有効にしないでください。

## セキュリティプラグインの役割

セキュリティプラグインは、WordPressの中で動作し、ログイン保護やスキャンなどを担います。WAFだけでは守れない範囲を、補完する役割があります。

セキュリティプラグインが得意とするのは、WordPress固有の保護です。ログインの保護、ユーザー名の露出防止、ファイルの変更検知など、WordPressの仕組みに即した機能を提供します。これらは、汎用的なWAFでは対応しにくい領域です。

セキュリティプラグインが主に担う範囲は、次のとおりです。WordPressに特化した保護が中心です。

- **ログイン保護**：ログイン試行回数の制限、二要素認証など
- **ユーザー管理**：admin名の検出、ユーザー名の露出防止
- **ファイル保護**：ファイルの変更検知、権限の確認
- **スキャン**：マルウェアの検出

セキュリティプラグインの中には、WAFを備えた総合型もあります。この場合、サーバーやCDNのWAFと重複しないよう注意してください。総合型を使う場合は、WAFの役割をどこが担うかを、明確にする必要があります。

なお、防御系のセキュリティプラグインは、1つに絞ってください。複数の防御系プラグインを併用すると、ログイン保護などが競合し、管理画面に入れなくなることがあります。

## CDNの役割

CDNは、本来はサイトの配信を高速化するためのサービスですが、一部のCDNはセキュリティ機能も提供します。サイトに到達する前の段階で、攻撃を防げる場合があります。

CDNは、サイトのコンテンツを世界中のサーバーに分散して配信します。この仕組みは、大量のアクセスを分散させるため、一部の攻撃への耐性にもなります。また、CDNによっては、WAFやアクセス制限の機能を提供するものもあります。

CDNが提供しうるセキュリティ機能は、次のとおりです。サービスによって、提供内容が異なります。

- **WAF**：一部のCDNが、攻撃パターンの遮断機能を提供します
- **大量アクセスへの耐性**：アクセスを分散させることで、一部の攻撃を緩和します
- **アクセス制限**：地域やIPによる制限を提供するものがあります

CDNを使う場合、そのCDNがどのセキュリティ機能を提供するかを確認してください。CDNでWAFを担うなら、サーバーやプラグインのWAFは不要になります。ここでも、機能の重複を避けることが重要です。

ただし、すべてのサイトにCDNが必要なわけではありません。小規模なサイトでは、CDNを導入せず、サーバーとプラグインで防御を組む選択も十分に成立します。

## サーバー標準機能の役割

サーバー標準機能は、レンタルサーバーが標準で提供する保護です。基盤に近い位置で動作し、多くの場合、追加の設定なしで一定の保護を提供します。

サーバー標準機能には、WAFのほか、様々な保護が含まれることがあります。これらは、サーバー事業者が管理するため、利用者の手間が少ないという利点があります。

サーバーが標準で提供しうる機能は、次のとおりです。サーバーによって、提供内容が異なります。

- **WAF**：多くのサーバーが標準で提供します
- **国外IP制限**：海外からのアクセスを遮断する機能
- **不正アクセスの検知**：一定のパターンに合致するアクセスを遮断します
- **SSL証明書**：通信の暗号化

サーバー標準機能は、まず確認すべき防御の土台です。すでに提供されている機能を、プラグインで重ねて用意する必要はありません。サーバーがWAFを提供しているなら、それを活かし、プラグイン側では別の役割を担わせてください。

## 重複と抜けを点検する

各層の役割を理解したら、自分のサイトの防御に、重複や抜けがないかを点検してください。重複は無駄と誤検知を生み、抜けは侵入口になります。

点検の際は、機能ごとに、どの層が担っているかを確認します。次のような点検表を作ると、重複と抜けが見えてきます。

| 機能 | サーバー | プラグイン | CDN | 状態 |
|---|---|---|---|---|
| WAF | | | | 1つが担当。重複していないか |
| ログイン保護 | | | | プラグインが担当が一般的 |
| SSL | | | | どこかで必ず担当 |
| バックアップ | | | | 外部保存を確保 |
| マルウェアスキャン | | | | 抜けていないか |

この表で、各機能について、どの層が担っているかを埋めてください。埋めていくと、2つの問題が見えます。1つは、同じ機能を複数の層が担っている重複。もう1つは、どの層も担っていない抜けです。

WAFのように重複しやすい機能は、1つの層に絞ってください。逆に、マルウェアスキャンやバックアップのように、抜けやすい機能は、どこかの層で必ず担うようにします。この点検を定期的に行うことで、防御の過不足を防げます。

## 抜けが生じやすい機能

防御の役割分担で、とくに抜けが生じやすい機能があります。これらは、どの層も「他が担っているだろう」と考えて、結果的にどこも担わない、という事態が起こりがちです。

抜けやすい機能と、その理由は次のとおりです。意識して確認しないと、見落とされます。

- **バックアップ**：防御ではないため、防御を組む際に忘れられがちです。しかし、最後の復旧手段として必須です
- **マルウェアスキャン**：WAFやログイン保護に注目が集まり、侵害後の検知が抜けることがあります
- **ファイルの変更検知**：改ざんに気づく手段ですが、導入されないことが多くあります
- **更新の継続**：プラグインやサーバー機能に頼り、肝心の更新が滞ることがあります

とくに、最後の更新の継続は重要です。WAFやセキュリティプラグインを導入すると、それだけで守られていると錯覚し、更新をおろそかにすることがあります。しかし、脆弱性そのものを塞ぐ更新は、防御の基本です。どんな防御を組んでも、更新の継続は欠かせません。

これらの抜けやすい機能は、防御の各層とは別に、意識して確保してください。派手なWAFやファイアウォールに注目が集まりがちですが、地味な更新やバックアップ、検知の仕組みこそが、実際の被害を防ぎます。

## 役割分担の基本的な考え方

複数の層をどう組み合わせるかに、唯一の正解はありません。ただし、基本的な考え方はあります。土台から順に、抜けを埋めていく形です。

役割分担を考える際の、基本的な順序は次のとおりです。この順で、防御を組み立ててください。

1. **サーバー標準機能を確認する**：まず、サーバーが何を提供しているかを確認します。これが土台です
2. **抜けをプラグインで埋める**：サーバーで足りない部分を、セキュリティプラグインで補います
3. **バックアップを別に確保する**：どの層でも、外部保存のバックアップは別に用意します
4. **必要ならCDNを加える**：規模や要件に応じて、CDNを検討します

この順序の要点は、土台となるサーバー機能から確認し、その抜けを埋めていくことです。プラグインをやみくもに増やすのではなく、サーバーで足りない部分だけを補う。この考え方が、重複を避けつつ抜けをなくす鍵になります。

そして、どの構成でも、バックアップは必ず確保してください。バックアップは防御ではなく復旧の手段ですが、防御をすり抜けられたときの最後の備えです。防御の役割分担を考える際も、この復旧手段を忘れないでください。

## WordPress側の設定を確認する

防御の役割分担を整理すると同時に、WordPress側の基本設定も確認してください。層を組み合わせても、WordPress自体の設定が甘ければ、そこが弱点になります。

## よくある質問

**Q. WAFは、どこで提供されるものを使うべきですか。**
A. サーバーが提供しているなら、それを使うのが手軽です。重要なのは、複数のWAFを重複させないことです。1つの層で担ってください。

**Q. セキュリティプラグインとサーバー機能は、両方必要ですか。**
A. 役割が異なるため、組み合わせます。ただし、WAFのように重複する機能は、一方に絞ってください。

**Q. CDNは必要ですか。**
A. すべてのサイトに必要なわけではありません。小規模なサイトは、サーバーとプラグインで十分な場合が多くあります。

**Q. 防御を組んでも、バックアップは必要ですか。**
A. 必要です。バックアップは、防御をすり抜けられたときの最後の復旧手段です。防御とは別に、必ず確保してください。

## まとめ

WordPressの防御は、WAF、セキュリティプラグイン、CDN、サーバー標準機能という複数の層で担われます。これらは守る範囲が一部重なり、一部は抜けています。重複させると誤検知や無駄が生じ、抜けがあればそこが侵入口になります。役割分担の基本は、まずサーバー標準機能を確認し、その抜けをプラグインで埋め、バックアップを別に確保することです。とくに、WAFは複数の層で提供されるため、1つに絞ってください。機能ごとにどの層が担うかを点検表で整理すれば、重複と抜けが見えます。この点検を定期的に行い、過不足のない防御を組んでください。
