# WordPress ECサイトセキュリティ｜決済と個人情報

WordPressでECサイトを運営する場合、守るべき情報は決済情報と顧客情報です。この2つは、漏えいすると被害が大きく、法的な責任も伴います。重要なのは、決済情報を自サイトで保持しない設計にすることです。この記事では、決済情報・顧客情報・管理者権限・拡張機能の4点で確認すべき要件を整理し、決済代行サービスとの責任分担を明確にします。一般のサイトとの違いに注目し、ECサイト特有のリスクに絞って説明します。

## ECサイトで守るべきものは何か

WordPressのECサイトで守るべき情報は、決済情報と顧客情報です。通常のサイトと違い、金銭と個人情報を直接扱うため、侵害されたときの被害が格段に大きくなります。

ECサイトが扱う情報は、攻撃者にとって価値が高いものです。クレジットカード情報は直接金銭につながり、顧客の個人情報は別の詐欺にも悪用されます。攻撃者にとって、労力に見合う見返りが得られる標的だということです。そのため、一般的なサイトより狙われやすく、対策の水準も高める必要があります。

ECサイトのセキュリティで確認すべき点は、4つに整理できます。それぞれ、守る対象と対策が異なります。

| 確認項目 | 守る対象 | 中心となる対策 |
|---|---|---|
| 決済情報 | クレジットカード情報 | 自サイトで保持しない設計 |
| 顧客情報 | 個人情報、注文履歴 | 保持の最小化とアクセス制御 |
| 管理者権限 | サイトの操作権限 | 権限の分離と最小化 |
| 拡張機能 | 決済・カート等のプラグイン | 信頼できる拡張機能の選択 |

このうち、最も重要なのが決済情報の扱いです。決済情報を自サイトで保持するかどうかで、必要な対策の水準が大きく変わります。以降、4点それぞれを説明します。

## ECサイトが狙われやすい理由

ECサイトへの対策を考える前に、なぜECサイトが特に狙われるのかを理解しておくと、対策の優先順位が見えてきます。一般のサイトとは、攻撃者の動機が異なります。

一般のサイトが乗っ取られる場合、その目的はスパムの送信や、別の攻撃の踏み台にすることが中心です。しかしECサイトの場合、攻撃者は直接的な金銭を狙います。この動機の違いが、攻撃の執拗さにつながります。

ECサイトが狙われる主な理由は、次のとおりです。いずれも、扱う情報の価値の高さに起因します。

- **決済情報が直接金銭になる**：クレジットカード情報は、そのまま不正利用や転売の対象になります
- **顧客情報が豊富**：氏名、住所、電話番号、購入履歴といった情報がまとまって存在します
- **偽の決済画面を仕込める**：改ざんして偽の決済画面を設置すれば、顧客のカード情報を直接盗めます

とくに3つ目は、ECサイト特有の攻撃です。サイトを改ざんし、正規の決済画面に見せかけた偽の画面へ誘導する手口です。この場合、決済代行を使っていても、顧客が偽画面にカード情報を入力してしまう被害が起こりえます。

この攻撃を防ぐには、そもそもサイトを改ざんされないことが重要です。改ざんの検知と、改ざんの前提となる侵入の防止が、ECサイトでは特に重要になります。扱う情報の価値が高いぶん、一般のサイトより高い水準の対策が求められます。

## 決済情報は自サイトで保持しない

ECサイトのセキュリティで最も重要な判断は、決済情報を自サイトで保持しないことです。クレジットカード情報を自分のサーバーに保存しない設計にすれば、万一侵害されても、カード情報の漏えいを避けられます。

決済情報を自サイトで保持すると、その保護に高い水準が求められます。クレジットカード情報の取り扱いには、業界の厳格な基準があり、それを満たすには大きな負担が伴います。個人や中小規模の事業者が、この水準を自前で維持するのは現実的ではありません。

現実的な解決策は、決済を外部の決済代行サービスに任せることです。この方式では、顧客がカード情報を入力する画面は決済代行側が用意し、カード情報は自サイトを経由しません。次のような違いがあります。

- **自サイトで保持する場合**：カード情報が自サーバーに保存される。高い水準の保護と、業界基準への対応が必要
- **決済代行に任せる場合**：カード情報は決済代行側で処理される。自サイトはカード情報を持たない

決済代行に任せる方式では、カード情報が自サイトを通らないため、侵害されてもカード情報は漏れません。ECサイトを構築する際は、この方式を選ぶのが基本です。カード情報を自前で扱う設計は、避けてください。

## 決済代行との責任分担を理解する

決済を外部に任せる場合でも、すべての責任が決済代行側に移るわけではありません。どこまでが決済代行の責任で、どこからが自分の責任かを理解しておく必要があります。

決済代行を使う場合の、おおまかな責任分担は次のとおりです。境界を理解しないと、自分が守るべき部分が抜けます。

| 領域 | 主な責任 |
|---|---|
| カード情報の処理・保管 | 決済代行サービス |
| 決済画面への正しい誘導 | 自サイト |
| 顧客情報（氏名・住所等）の管理 | 自サイト |
| サイト全体のセキュリティ | 自サイト |

カード情報そのものは決済代行が扱いますが、顧客をその決済画面へ正しく誘導するのは自サイトの役割です。サイトが改ざんされ、偽の決済画面へ誘導されれば、決済代行を使っていても被害が生じます。

また、氏名や住所、注文履歴といった顧客情報は、自サイトで管理します。カード情報以外の個人情報は、依然として自分の責任範囲です。決済代行に任せたからといって、サイト全体のセキュリティが不要になるわけではありません。

## 顧客情報の保持を最小限にする

決済情報の次に重要なのが、顧客情報の扱いです。氏名、住所、電話番号、注文履歴といった情報は、自サイトで管理します。この情報を守る基本は、保持する量を最小限にすることです。

保持していない情報は、漏れようがありません。ECサイトの運営に本当に必要な情報だけを保持し、不要な情報は集めない、あるいは保持し続けないことが、漏えいリスクを下げます。

顧客情報の保持について、確認すべき点は次のとおりです。何を、どれだけ持つかを見直してください。

- **必要な情報だけを集める**：注文処理に不要な情報は、そもそも入力させないことを検討します
- **保持期間を決める**：古い注文情報を、いつまでも保持し続けないようにします
- **アクセスできる人を限定する**：顧客情報を閲覧できる管理者を、必要な人だけにします
- **バックアップにも注意する**：バックアップにも顧客情報が含まれます。バックアップの保存先も安全に管理します

とくに、アクセスできる人の限定は重要です。顧客情報を閲覧できる権限を、運営に必要な最小限の人だけに絞ってください。管理者権限を持つ人が多いほど、情報が漏れる経路も増えます。

個人情報の漏えいが発生した場合、状況によっては関係機関への報告や本人への通知が必要になることがあります。具体的な対応は、公的な情報や専門家の判断を確認してください。

## 管理者権限を分離する

ECサイトでは、通常のサイト以上に、管理者権限の管理が重要になります。管理者権限を持つアカウントが乗っ取られると、顧客情報へのアクセスや、決済画面の改ざんにつながるためです。

ECサイトの運営には、複数の人が関わることがあります。商品を登録する人、注文を処理する人、サイトを管理する人。それぞれに、全権限を与える必要はありません。役割に応じた権限を割り当ててください。

権限を分離する際の考え方は、次のとおりです。誰が何にアクセスできるかを、最小限にします。

- **役割ごとに権限を分ける**：商品登録、注文処理、サイト管理で、必要な権限は異なります
- **顧客情報へのアクセスを絞る**：顧客情報を見る必要がある人だけに、その権限を与えます
- **管理者を最小限にする**：すべての操作ができる管理者は、できるだけ少なくします

WooCommerceなどのEC向け拡張機能には、店舗運営用の権限グループが用意されていることがあります。これを活用すると、全権限を与えずに、必要な操作だけを許可できます。

管理者アカウントには、二要素認証を導入してください。乗っ取られたときの被害が最も大きいため、認証を強化する価値が高くなります。

## 拡張機能を厳選する

ECサイトは、カート機能や決済機能を、拡張機能（プラグイン）で実現します。これらの拡張機能に脆弱性があると、決済や顧客情報に関わる重大な被害につながります。拡張機能の選択は、慎重に行ってください。

ECサイトで使う拡張機能は、扱う情報が重要なため、一般的なプラグイン以上に信頼性が求められます。選ぶ際の基準は、次のとおりです。

- **保守されているか**：最終更新日を確認し、継続的に保守されているものを選びます
- **実績があるか**：多くのECサイトで使われ、問題が発見・報告されてきたものを選びます
- **決済関連は特に慎重に**：決済に関わる拡張機能は、公式や信頼できる開発元のものに限定します
- **数を最小限にする**：機能を欲張らず、必要な拡張機能だけに絞ります

とくに、決済に関わる拡張機能は、慎重に選んでください。無償で配布されている出所不明の決済プラグインには、不正なコードが含まれている可能性があります。公式や、決済代行サービスが提供する正規の拡張機能を使ってください。

拡張機能の数を最小限にすることも重要です。ECサイトは機能が多くなりがちですが、拡張機能が増えるほど、脆弱性を抱える箇所も増えます。本当に必要な機能だけに絞ってください。

## SSL証明書は必須

ECサイトでは、SSL証明書による通信の暗号化が必須です。顧客が個人情報や、決済に関わる情報を入力するため、その通信が暗号化されていないと、情報が読み取られる危険があります。

一般のサイトでもSSL化は推奨されますが、ECサイトでは必須の要件です。顧客が入力する情報の重要性が、格段に高いためです。暗号化されていない通信では、入力された情報が第三者に盗み見られる可能性があります。

SSL証明書について、ECサイトで確認すべき点は次のとおりです。サイト全体での適用が前提になります。

- **サイト全体をHTTPSにする**：一部のページだけでなく、サイト全体を暗号化します
- **証明書の有効期限を管理する**：期限切れになると、警告が表示され、信頼を損ないます
- **混在コンテンツをなくす**：HTTPSのページ内に、暗号化されていない要素が残らないようにします

多くのレンタルサーバーが、無料のSSL証明書を提供しています。ECサイトを構築する際は、必ずSSL化を行い、サイト全体を暗号化してください。SSL化されていないECサイトは、顧客からの信頼も得られません。

## 不正注文への備え

ECサイト特有のリスクとして、不正注文があります。盗まれたカード情報を使った注文や、いたずらの大量注文などです。これらは、セキュリティ対策とは別に、運用面での備えが必要です。

不正注文への備えは、決済代行サービスの機能と、自サイトの運用の両面で行います。主な対策は次のとおりです。

- **決済代行の不正検知機能**：多くの決済代行サービスが、不正な取引を検知する機能を提供しています。これを活用します
- **注文内容の確認**：高額な注文や、不自然な注文について、発送前に確認する運用を設けます
- **ログイン保護**：会員機能がある場合、アカウントの乗っ取りを防ぐログイン保護を導入します

不正注文は、完全には防げません。決済代行の機能を活用しつつ、不自然な注文に気づける運用を整えることで、被害を減らせます。

不正注文を放置すると、正規の顧客とのトラブルや、チャージバック（不正利用されたカードの利用者からの返金請求）による損失につながります。とくに、盗まれたカード情報を使った注文は、後から返金請求が来ることがあり、商品を発送してしまうと二重の損失になります。高額な注文や、届け先と請求先が異なる注文などは、発送前に確認する体制を設けてください。

## ECサイトのセキュリティ要件を一覧にする

ここまでの内容を、ECサイトを構築・運用する際の要件として一覧にまとめます。設計時と運用時に、この一覧で抜けがないかを確認してください。

ECサイトのセキュリティ要件を、設計時と運用時に分けて整理します。それぞれの段階で、確認すべき項目が異なります。

| 段階 | 要件 |
|---|---|
| 設計時 | 決済代行の選定（カード情報を自サイトで持たない） |
| 設計時 | 顧客情報の保持を最小限にする設計 |
| 設計時 | 権限の分離（役割ごとの権限設定） |
| 設計時 | SSL証明書によるサイト全体の暗号化 |
| 運用時 | 拡張機能の更新と棚卸し |
| 運用時 | 管理者アカウントの二要素認証 |
| 運用時 | 不正注文の確認体制 |
| 運用時 | バックアップと復元の確認 |

設計時の要件は、運用開始前に固めておくものです。とくに決済代行の選定と、顧客情報の保持設計は、後から変更するのが困難です。最初に決めてください。

運用時の要件は、継続して行うものです。拡張機能の更新や、不正注文の確認は、日々の運用として回します。この一覧を、ECサイトのセキュリティチェックリストとして使ってください。

ECサイトは、扱う情報の重要性から、一般のサイトより高い水準の対策が求められます。しかし、要件を整理して1つずつ対応すれば、過度に複雑なものではありません。決済情報を自サイトで持たない設計を軸に、顧客情報を守る体制を整えてください。

## WordPress側の基本設定も確認する

ECサイト特有の対策と並行して、WordPressの基本的なセキュリティ設定も確認してください。土台となる設定が甘ければ、EC特有の対策も活きません。

## よくある質問

**Q. 決済情報は自分で管理してもいいですか。**
A. 推奨しません。高い水準の保護と業界基準への対応が必要で、負担が大きくなります。決済代行に任せ、カード情報を自サイトで保持しない設計にしてください。

**Q. 決済代行を使えば、セキュリティ対策は不要ですか。**
A. 不要にはなりません。顧客情報の管理やサイト全体のセキュリティは、自分の責任です。決済代行が守るのはカード情報の処理部分です。

**Q. ECサイトは一般のサイトより狙われますか。**
A. 狙われやすくなります。決済情報や顧客情報は攻撃者にとって価値が高いためです。対策の水準を高める必要があります。

**Q. 顧客情報はどこまで集めるべきですか。**
A. 注文処理に必要な最小限に絞ってください。保持していない情報は漏れません。不要な情報は集めない設計が安全です。

## まとめ

WordPressのECサイトで守るべきは、決済情報と顧客情報です。最も重要なのは、決済情報を自サイトで保持しない設計にすることです。決済を外部の決済代行サービスに任せ、カード情報が自サイトを経由しないようにしてください。ただし、顧客情報の管理やサイト全体のセキュリティは、依然として自分の責任です。顧客情報は保持を最小限にし、管理者権限を分離し、拡張機能を厳選してください。決済代行との責任分担を理解し、自分が守るべき範囲を明確にすることが、ECサイトのセキュリティの出発点になります。扱う情報の価値が高いぶん、設計段階から慎重に要件を固めてください。
