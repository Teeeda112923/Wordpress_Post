# WordPressセキュリティ設定20項目チェック

WordPressのセキュリティ設定は、20項目に整理できます。多くのサイトで抜けているのは、既定の管理者名、ファイル編集機能、デバッグ表示、ユーザー名の露出の4つです。すべてを一度にやる必要はありません。影響の大きい順に確認し、抜けている箇所だけを直してください。この記事では、各項目の確認場所と設定方法を、管理画面の操作単位で示します。

## 確認は5つの領域に分かれる

WordPressのセキュリティ設定は、アカウント・ログイン・ファイル・公開情報・運用の5領域に分けると整理できます。領域ごとに確認する画面が違うため、この順で見ていくと作業が最短で終わります。

領域と項目数の内訳は次のとおりです。上の領域ほど、侵入に直結しやすい設定が集まっています。

| 領域 | 項目数 | 主に確認する場所 |
|---|---|---|
| アカウントと権限 | 5項目 | 管理画面「ユーザー」 |
| ログイン保護 | 4項目 | セキュリティプラグインの設定画面 |
| ファイルと設定値 | 5項目 | wp-config.php、サーバーのファイル管理画面 |
| 公開情報と機能 | 4項目 | ブラウザ、wp-config.php |
| 通信と運用 | 2項目 | サーバー管理画面、バックアップ設定 |

作業に入る前に、バックアップを取得してください。確認するだけならサイトに影響はありませんが、設定を変更する段階では、戻せる状態を作っておく必要があります。

## 設定を変更する前に確認すること

20項目のうち、いくつかは変更によってサイトが動かなくなる可能性があります。作業に入る前に、次の3点を必ず押さえてください。ここを飛ばすと、直すつもりの作業が障害対応に変わります。

特に注意が必要なのは、ログイン保護とファイル権限に関する項目です。設定を誤ると、管理者自身が管理画面に入れなくなります。

- **バックアップを取得する**：ファイルとデータベースの両方。保存先はサーバー内ではなく外部に置きます
- **1つずつ変更し、都度確認する**：複数の設定をまとめて変えると、不具合が出たときに原因を特定できません。1項目変更したら、サイトの表示と管理画面へのログインを確認します
- **締め出されたときの復旧手段を用意する**：FTPまたはサーバーのファイルマネージャーへの接続情報を、事前に手元で確認しておきます

3つ目を軽視しないでください。ログインURLの変更やIP制限で自分が締め出された場合、復旧はサーバー側からの操作になります。接続情報が分からないまま作業を始めると、そこから調べることになります。

wp-config.php を編集する項目については、編集前にファイルをダウンロードして保管してください。問題が起きた場合、元のファイルに戻すだけで復旧できます。

## アカウントと権限（5項目）

侵入経路として最も使われるのが、認証情報です。ここが緩いままだと、他の設定を固めても意味が薄くなります。

### 1. 管理者ユーザー名が admin になっていない

管理画面の「ユーザー」を開き、権限グループが「管理者」のアカウント名を確認します。admin または administrator が存在する場合、攻撃者が最初に試す名前と一致しています。

WordPressはユーザー名の直接変更に対応していません。新しい名前の管理者アカウントを作成し、投稿の所有者を引き継いだうえで、既存のアカウントを削除します。この作業では投稿が失われる可能性があるため、必ずバックアップを取ってから行ってください。

### 2. パスワードを使い回していない

他のサービスから流出したパスワードを試す攻撃は、成功率が高く多用されています。WordPressのパスワード生成機能で20文字以上のランダムな文字列を設定し、パスワード管理ツールに保存してください。記憶する必要はありません。

### 3. 管理者権限のアカウントが最小限になっている

管理者権限は、必要な人だけに与えます。記事を書くだけであれば「編集者」や「投稿者」で足ります。管理者アカウントが多いほど、乗っ取られる確率は上がります。

### 4. 不要なアカウントが残っていない

運用が長くなると、退職した担当者や制作会社のアカウントが管理者のまま残っていることがあります。使われていないアカウントは削除してください。半年に一度、ユーザー一覧を確認する習慣をつけます。

### 5. 権限グループが適切に割り当てられている

WordPressの権限は、管理者・編集者・投稿者・寄稿者・購読者に分かれます。それぞれ実行できる操作が違います。全員を管理者にしている運用は、事故と侵害の両方のリスクを高めます。

## ログイン保護（4項目）

ログイン画面には、日常的にボットからの試行が届いています。以下の4項目は、いずれもセキュリティプラグインで設定できます。

### 6. ログイン試行回数が制限されている

既定のWordPressは、ログイン試行を無制限に受け付けます。一定回数の失敗でIPアドレスを一時的に遮断する設定を入れてください。これだけで、総当たり攻撃の大半は成立しなくなります。

### 7. 二要素認証が有効になっている

パスワードが漏れても、二要素認証があれば侵入を防げます。導入時に発行される復旧コードは、必ず別の場所に保管してください。端末を紛失した際、これがないと自分がログインできなくなります。

### 8. 管理画面へのアクセスが制限されている

固定IPから作業している場合は、管理画面（/wp-admin）へのIPアドレス制限が最も確実です。固定IPがない場合は、ログインURLの変更やCAPTCHAで代替します。ただしURLの変更は攻撃を見えにくくするだけで、根本的な防御ではありません。

### 9. ログインエラーの詳細が表示されない

既定では、ユーザー名が間違っている場合とパスワードが間違っている場合で、異なるエラーメッセージが表示されます。これは、ユーザー名が存在するかどうかを攻撃者に教えることになります。エラーメッセージを共通化する設定を有効にしてください。

## ファイルと設定値（5項目）

ここからは、wp-config.php とサーバー側の設定です。ファイルを編集する作業が含まれるため、変更前に必ずバックアップを取得してください。

### 10. 管理画面からのファイル編集が無効になっている

WordPressの管理画面には、テーマやプラグインのコードを直接書き換えられる機能があります。管理者アカウントが乗っ取られた場合、この機能で任意のプログラムを実行されます。WordPress公式のハードニング文書でも無効化が推奨されています。

wp-config.php に次の行を追加してください。

```
define('DISALLOW_FILE_EDIT', true);
```

### 11. デバッグ表示が無効になっている

wp-config.php の `WP_DEBUG` が有効だと、エラー内容が画面に表示されます。サーバー内部のファイルの場所などが含まれるため、本番環境では無効にしてください。開発中に有効にして、公開時に戻し忘れるのが典型的なパターンです。

### 12. ファイル権限が適切に設定されている

一般的な推奨値は、ディレクトリが755、ファイルが644です。wp-config.php は、さらに厳しい権限（600など）に設定できます。ただし、サーバーの構成によって適切な値は異なるため、変更してサイトが動かなくなった場合は元に戻してください。

### 13. wp-config.php が保護されている

このファイルには、データベースの接続情報が記述されています。外部から読み取られると、データベースへ直接アクセスされる可能性があります。Web公開ディレクトリの外に移動するか、サーバー設定でアクセスを禁止してください。

### 14. データベース接頭辞が既定値でない

既定値は `wp_` です。このままだと、SQLインジェクションの脆弱性があった場合に、攻撃者がテーブル名を推測しやすくなります。

ただし、稼働中のサイトで接頭辞を変更する作業はリスクが高く、失敗するとサイトが表示されなくなります。**新規構築時に変更しておくのが本来の対処**であり、既存サイトで無理に変更する必要性は高くありません。

## 公開情報と機能（4項目）

外部から取得できる情報を減らす設定です。単体では被害に直結しませんが、攻撃者に手がかりを与えます。

### 15. REST API でユーザー名が取得できない

ブラウザで `https://（自分のサイト）/wp-json/wp/v2/users` を開いてください。ユーザー情報が表示される場合、ログインIDが外部から取得できる状態です。総当たり攻撃の難易度が半分になります。

未認証でのユーザー一覧取得を制限する設定を追加してください。REST API 自体は多くのプラグインが利用しているため、全体を無効化するのではなく、ユーザー列挙の部分だけを塞ぐのが安全です。

### 16. XML-RPC の要否を判断している

XML-RPC は、外部アプリケーションからWordPressを操作するための仕組みです。有効な状態では、この経路から認証を試行できます。1回のリクエストで複数のパスワードを試せるため、ログイン試行回数の制限を迂回されることがあります。

ただし、Jetpack など一部のプラグインや外部連携ツールがこの仕組みを使っています。使っていないなら無効化し、使っているならそのままにしてください。使用状況を確認せずに無効化すると、連携機能が止まります。

### 17. バージョン情報が露出していない

サイトのソースを表示し、`generator` を検索してください。WordPressのバージョン番号が記載されていれば、外部に公開されています。あわせて `https://（自分のサイト）/readme.html` にアクセスし、ページが表示されるかも確認します。

バージョンを隠すこと自体は根本的な対策ではありませんが、自動化されたスキャンの対象から外れる効果はあります。

### 18. ディレクトリの一覧表示が無効になっている

`https://（自分のサイト）/wp-content/uploads/` にアクセスして、ファイルの一覧が表示される場合、サーバーの設定でディレクトリ一覧表示が有効になっています。アップロードしたファイルがすべて見える状態は望ましくありません。サーバーの設定で無効化してください。

## 通信と運用（2項目）

### 19. 常時HTTPSになっている

サイトのURLが `http://` で始まる場合、ログイン時のパスワードが暗号化されずに送信される可能性があります。SSL証明書を適用し、すべてのページをHTTPSに統一してください。多くのレンタルサーバーが、無料のSSL証明書を提供しています。

### 20. バックアップを取得し、復元を確認している

セキュリティ設定の最後に置いていますが、実務上の重要度は最も高い項目です。ファイル一式とデータベースの両方を取得し、**サーバー内ではなく外部ストレージに保存**してください。サーバーごと侵害されれば、サーバー内のバックアップも同時に失われます。

そして、実際に復元できるかを一度試してください。取得しているだけで復元を試したことがなければ、いざというときに動きません。

## 20項目を影響の大きい順に並べ替える

すべてを一度にやる必要はありません。時間が限られている場合は、次の順で処理してください。

上位の項目ほど、少ない手間で大きくリスクを下げられます。

| 優先度 | 項目 | 想定工数 |
|---|---|---|
| 高 | 20（バックアップ）、2（パスワード）、10（ファイル編集の無効化）、11（デバッグ表示）、19（HTTPS） | 各10〜60分 |
| 中 | 6（ログイン試行回数）、7（二要素認証）、1（管理者名）、4（不要アカウント）、15（ユーザー名の露出） | 各20〜40分 |
| 低 | 8（管理画面の制限）、9（エラーメッセージ）、16（XML-RPC）、17（バージョン露出）、18（ディレクトリ一覧） | 各10〜30分 |
| 環境依存 | 12（ファイル権限）、13（wp-config保護）、14（DB接頭辞）、3・5（権限設計） | 要検証 |

高の5項目だけでも、今日中に終わります。この5つを片付ければ、自動化された攻撃に対する耐性は大きく変わります。

環境依存に分類した項目は、変更によってサイトが動かなくなる可能性があります。バックアップを取得し、1つずつ変更して都度確認してください。

## 設定の抜けを一覧で確認する

## よくある質問

**Q. 20項目すべてを設定する必要がありますか。**
A. 必要ありません。優先度が高い5項目を先に処理してください。環境依存の項目は、無理に変更しないほうが安全な場合があります。

**Q. wp-config.php を編集するのが不安です。**
A. 編集前に必ずファイルをダウンロードして保管してください。問題が起きた場合、元のファイルに戻せば復旧できます。

**Q. XML-RPC は無効にすべきですか。**
A. 使っていなければ無効化してください。外部連携ツールやアプリで使用している場合は、そのままにします。

**Q. データベース接頭辞は変更したほうがいいですか。**
A. 新規構築時であれば変更してください。稼働中のサイトでは、作業リスクのほうが大きくなる場合があります。

## まとめ

WordPressのセキュリティ設定は、アカウント・ログイン・ファイル・公開情報・運用の5領域、20項目に整理できます。多くのサイトで抜けているのは、既定の管理者名、管理画面からのファイル編集機能、デバッグ表示、REST APIによるユーザー名の露出です。すべてを一度に設定する必要はありません。バックアップ、パスワード、ファイル編集の無効化、デバッグ表示の停止、HTTPS化。この5つを先に済ませてください。残りは、影響範囲を確認しながら1つずつ進めれば十分です。
