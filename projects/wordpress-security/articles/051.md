# WordPress REST APIセキュリティ｜公開範囲の判断

WordPressのREST APIは、完全に無効化すべきではありません。無効化すると、ブロックエディタや多くのプラグインが動かなくなるためです。必要なのは、公開範囲を見極め、制限すべき部分だけを制御することです。この記事では、REST APIのどこにリスクがあるかを整理し、ユーザー情報の露出やアップロードファイルの一覧といった、制限すべき範囲の判断方法を示します。連携機能を壊さずに安全性を高める方法を説明します。

## REST APIを完全に無効化してはいけない

WordPressのREST APIのセキュリティを考えるとき、最初に理解すべきは、完全な無効化をしてはいけないという点です。REST APIを全面的に止めると、WordPressの多くの機能が動かなくなります。

REST APIは、外部のプログラムからWordPressのデータをやり取りするための仕組みです。しかし、外部連携のためだけのものではありません。WordPress自身の管理画面、とくに投稿を編集するブロックエディタが、この仕組みを使って動いています。

REST APIを完全に無効化すると、次のような影響が出ます。「危険だから止める」という判断が、かえって支障を生みます。

- **ブロックエディタが動かなくなる**：投稿の編集ができなくなります
- **多くのプラグインが機能しなくなる**：REST APIを利用するプラグインが動きません
- **サイトの一部機能が停止する**：テーマやプラグインの一部の機能が使えなくなります

したがって、REST APIのセキュリティ対策は、「無効化する」ことではありません。「どの部分に、どんなリスクがあるか」を見極め、制限すべき範囲だけを制御することです。以降、リスクのある部分を整理します。

## REST APIのどこにリスクがあるか

REST APIのリスクは、API全体ではなく、特定の情報が外部から取得できる点にあります。どの情報が、認証なしで取得できるのかを把握することが、対策の出発点です。

REST APIには、多数のエンドポイントがあります。エンドポイントとは、情報を取得するための入口のようなものです。この中で、認証なしでアクセスできるものと、認証が必要なものがあります。リスクがあるのは、認証なしで重要な情報が取得できる部分です。

とくに注意すべき情報は、次の2つです。いずれも、初期状態では認証なしで取得できることがあります。

| 取得できる情報 | リスク |
|---|---|
| ユーザー情報 | ユーザー名（ログインID）が外部に露出する |
| アップロードファイルの一覧 | 非公開のつもりのファイルのURLが露出する |

この2つが、REST APIで特に注意すべき情報です。API全体を止めるのではなく、この2つの露出を制御することが、現実的な対策になります。以降、それぞれを詳しく説明します。

## リスク1：ユーザー情報の露出

REST APIを通じて、WordPressに登録されているユーザーの情報が取得できることがあります。この中には、ユーザー名（ログインID）が含まれます。

ブラウザで `https://（自分のサイト）/wp-json/wp/v2/users` にアクセスすると、ユーザーの一覧が表示される場合があります。ここにユーザー名が含まれていると、攻撃者はログインIDを推測する必要がなくなります。総当たり攻撃の難易度が、大きく下がります。

とくに問題になるのは、ユーザー名に個人名を使っている場合です。ログインIDが本名だと、それが外部に露出することになります。次の点を確認してください。

- **ユーザー一覧が返るか**：`/wp-json/wp/v2/users` にアクセスして確認します
- **ユーザー名に何を使っているか**:本名や、推測しやすい名前を使っていないかを確認します
- **表示名とログインIDが同じか**：同じ場合、記事の投稿者欄からもログインIDが分かります

このユーザー情報の露出は、認証なしのユーザー一覧取得を制限することで防げます。セキュリティプラグインの多くが、この制限機能を提供しています。API全体を止めるのではなく、ユーザー一覧を返す部分だけを制限してください。

## リスク2：アップロードファイルの露出

REST APIは、サイトにアップロードされたファイルのURL一覧も取得できます。管理画面からアップロードした画像やPDFなどが、対象になります。

WordPressは、アップロードされたファイルを公開情報として扱います。そのため、ログインしていない状態でも、REST APIを通じてファイルのURL一覧を取得できることがあります。通常の画像であれば問題になりませんが、限られた人にだけURLを教えているファイルがある場合、そのURLが露出します。

アップロードファイルの露出が問題になるのは、次のようなケースです。ファイルの公開範囲を、URLの秘匿で管理している場合に注意が必要です。

- **限定配布のファイル**：特定の人にだけURLを教えているPDFなどが、一覧に含まれます
- **未公開の資料**：まだ公開していない資料をアップロードしている場合、そのURLが分かります
- **会員限定のコンテンツ**：URLを知られたくないファイルが、露出します

このようなファイルがある場合、REST APIのメディア情報を返すエンドポイントを制限する必要があります。ただし、この制限は、サイトの機能に影響することがあるため、慎重に判断してください。

そもそも、URLの秘匿だけでファイルを守る運用は、安全とは言えません。本当に守るべきファイルは、URLを知られない前提ではなく、アクセス制御の仕組みで守るべきです。REST APIの制限は、あくまで補助的な対策と考えてください。

## Application Passwordsを理解する

REST APIを外部から利用する際の認証方法として、Application Passwordsがあります。これは、WordPress本体に組み込まれた認証の仕組みです。外部連携を行う場合、この仕組みを正しく理解しておく必要があります。

Application Passwordsは、通常のログインパスワードとは別に、REST API接続専用のパスワードを発行する仕組みです。ユーザーごとに、用途別のパスワードを発行できます。外部のプログラムからWordPressを操作する際に使います。

Application Passwordsを使う際の注意点は、次のとおりです。認証情報の管理が、セキュリティの要になります。

- **HTTPS環境が必要**：Application Passwordsは、HTTPS環境でのみ使えます。暗号化されていない通信では使うべきではありません
- **用途ごとに発行する**：連携するプログラムごとに、別のパスワードを発行します。1つが漏れても、影響を限定できます
- **不要になったら削除する**：使わなくなったパスワードは、削除します
- **権限を最小限にする**：連携に必要な最小限の権限を持つユーザーで、パスワードを発行します

Application Passwordsは、発行したパスワードが漏れると、そのユーザーの権限でREST APIを操作されます。パスワードは慎重に管理し、不要になったものは削除してください。外部連携をしていない場合は、この機能を使う必要はありません。

## カスタムエンドポイントの注意点

プラグインやテーマが、独自のREST APIエンドポイントを追加することがあります。これをカスタムエンドポイントと呼びます。この実装に不備があると、脆弱性の原因になります。

カスタムエンドポイントの問題は、権限の確認が不十分な場合に起こります。本来アクセスできないはずの情報が、認証なしで取得できてしまう、あるいは権限のない操作が実行できてしまう、といった事態です。

これは、運営者が直接コードを書くわけではない場合、プラグインの選択の問題になります。次の点に注意してください。

- **信頼できるプラグインを選ぶ**：カスタムエンドポイントを追加するプラグインは、保守されている信頼できるものを選びます
- **脆弱性情報を確認する**：使っているプラグインに、REST API関連の脆弱性が報告されていないかを確認します
- **不要なものは削除する**：使っていないプラグインは削除し、不要なエンドポイントを減らします

REST API関連の脆弱性は、実際に多く報告されています。とくに、権限確認の不備は、深刻な情報漏えいにつながることがあります。プラグインの更新を継続し、脆弱性が報告されたものは速やかに対応してください。

## 制限すべき範囲を判断する

ここまでの内容を踏まえて、REST APIのどこを制限すべきかを判断します。全体を止めるのではなく、リスクのある部分だけを制御するのが基本です。

REST APIの制限について、判断の目安を整理します。自分のサイトの状況に合わせて、必要な制限を選んでください。

| 対象 | 制限すべきか |
|---|---|
| ユーザー一覧（認証なし） | 制限する（ほぼすべてのサイトで） |
| メディア一覧（認証なし） | 限定配布ファイルがある場合は制限 |
| REST API全体 | 制限しない（機能が壊れる） |
| 外部連携用の認証 | 使わないなら発行しない |

ユーザー一覧の制限は、ほぼすべてのサイトで行う価値があります。ユーザー名の露出を防ぐ、基本的な対策です。メディア一覧の制限は、限定配布のファイルがある場合に検討します。

一方、REST API全体の無効化は行わないでください。機能が壊れます。外部連携をしていない場合は、Application Passwordsを発行しないことで、その経路のリスクをなくせます。

## 現状のREST APIの状態を確認する

REST APIの制限を検討する前に、現在どの情報が露出しているかを確認してください。とくに、ユーザー情報の露出は、確認しておくべき項目です。

## よくある質問

**Q. REST APIは無効化したほうが安全ですか。**
A. いいえ。完全に無効化すると、ブロックエディタや多くのプラグインが動かなくなります。リスクのある部分だけを制限してください。

**Q. ユーザー情報の露出は、どう防ぎますか。**
A. 認証なしのユーザー一覧取得を制限します。セキュリティプラグインの多くが、この機能を提供しています。

**Q. Application Passwordsは使うべきですか。**
A. 外部連携をする場合に使います。していない場合は、発行する必要はありません。HTTPS環境でのみ使用してください。

**Q. アップロードしたファイルのURLが露出するのは問題ですか。**
A. 限定配布のファイルがある場合は問題です。ただし、URLの秘匿だけでファイルを守る運用は避け、アクセス制御で守るべきです。

## まとめ

WordPressのREST APIは、完全に無効化してはいけません。ブロックエディタや多くのプラグインが動かなくなるためです。対策は、公開範囲を見極め、制限すべき部分だけを制御することです。とくに注意すべきは、認証なしで取得できるユーザー情報と、アップロードファイルの一覧です。ユーザー一覧の制限は、ほぼすべてのサイトで行う価値があります。外部連携にはApplication PasswordsをHTTPS環境で使い、不要なパスワードは削除してください。全体を止めるのではなく、リスクのある範囲を見極めて制御することが、連携機能を壊さずに安全性を高める方法です。
