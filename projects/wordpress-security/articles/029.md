# WordPress脆弱性情報｜JVN・CVE・WPScan

WordPressの脆弱性情報は、情報源を4つに絞れば追いきれます。すべてを毎日眺める必要はありません。必要なのは、自分が使っているプラグインに関係する情報だけを拾い出す仕組みです。この記事では、JVN・CVE・WPScan・WordPress公式をどの順番で確認するか、影響バージョンと悪用情報をどう読むかを整理します。

## 情報源は4つに絞る

WordPressの脆弱性情報を集める際に使うのは、JVN iPedia、CVE／NVD、WPScan Vulnerability Database、WordPress公式のセキュリティリリース情報の4つです。それぞれ収録範囲と速報性が違うため、目的に応じて使い分けます。

すべてを網羅しようとすると続きません。まずは各情報源が何を得意としているかを把握してください。

| 情報源 | 運営 | 特徴 | 主な用途 |
|---|---|---|---|
| JVN iPedia | IPA・JPCERT/CC | 日本語で読める。国内向けの公的な情報源 | 社内報告、内容の正確な理解 |
| CVE／NVD | 米国の公的機関等 | 脆弱性の識別番号と深刻度の基準情報 | 影響範囲と深刻度の確認 |
| WPScan Vulnerability Database | WPScan | WordPress専門。プラグイン単位で検索できる | 特定のプラグインを調べる |
| WordPress公式リリース情報 | WordPress.org | 本体のセキュリティリリースを告知 | コアの更新要否の判断 |

日本語で確認したい場合はJVN iPedia、プラグイン単位で調べたい場合はWPScanが起点になります。CVEは、脆弱性ひとつひとつに割り当てられる識別番号です。同じ脆弱性が複数の情報源で報告されていても、CVE番号が一致していれば同一のものだと判断できます。

なお、これらの情報源が扱うのはWordPressエコシステム全体の脆弱性です。自分のサイトに関係のないものが大半を占めます。次章で、関係するものだけを選び出す方法を説明します。

## 自分に関係する情報だけを選び出す

公開される脆弱性の大半は、あなたのサイトとは無関係です。関係するかどうかは、製品名と影響バージョンの2点だけで判定できます。この2点を確認すれば、大半の情報はその場で切り捨てられます。

判定の手順は次のとおりです。上から順に見て、途中で条件から外れたら、その脆弱性はあなたのサイトに関係ありません。

1. **製品名を照合する**：導入しているプラグイン・テーマの名前と一致するか
2. **影響バージョンを照合する**：自分が使っているバージョンが、影響を受ける範囲に含まれるか
3. **修正版の有無を確認する**：修正版が公開されているか、まだ存在しないか
4. **悪用情報を確認する**：すでに攻撃が観測されているか

1と2で外れるものが圧倒的多数です。ここで機械的に振り分ければ、実際に読み込む必要がある情報は大きく減ります。

影響バージョンの表記には注意してください。「1.5.2未満」と書かれている場合、1.5.2そのものは修正済みです。「1.5.2以下」であれば1.5.2も影響を受けます。この違いを読み違えると、対応済みのものを再度直そうとしたり、逆に見逃したりします。

前提として、自分のサイトで何をどのバージョンで使っているかが分かっている必要があります。ここが曖昧なままでは、照合そのものが成立しません。

## セキュリティアドバイザリの読み方

脆弱性の公表情報は、セキュリティアドバイザリと呼ばれます。項目は決まっているため、どこを見ればよいかを覚えれば、読む時間は数分で済みます。

確認するのは次の5項目です。すべてを読み込む必要はなく、上から4つで対応の要否は判断できます。

- **製品名とバージョン**：対象と、影響を受けるバージョンの範囲
- **脆弱性の種類**：クロスサイトスクリプティング、権限昇格、任意のファイルアップロードなど。どんな被害につながるかが分かる
- **認証の要否**：ログインなしで悪用できるか。不要であれば、自動化された攻撃の対象になりやすい
- **修正版**：修正されたバージョン番号。記載がなければ、まだ修正されていない
- **CVSS**：深刻度を数値で示した指標

CVSSは0から10までの数値で、値が大きいほど深刻とされます。ただし、この数値だけで優先順位を決めないでください。深刻度が中程度でも、認証不要で悪用でき、自分が実際に使っているプラグインであれば、そちらが優先されます。

逆に、CVSSが高くても、悪用に管理者権限が必要な脆弱性であれば、緊急度は下がります。数値は判断材料の1つであって、結論ではありません。

## 悪用情報をどう扱うか

すでに攻撃が観測されている脆弱性は、最優先で対応します。悪用が確認されているということは、攻撃コードが出回っており、自動化されたスキャンの対象になっている可能性が高いためです。

悪用の有無は、脆弱性データベースの記載や、セキュリティ製品の提供元が公表する情報から確認できます。米国の公的機関が公表している、悪用が確認された脆弱性の一覧も参考になります。

ただし、悪用情報がないことは安全を意味しません。観測されていないだけで、実際には攻撃が始まっている可能性があります。公開から悪用開始までの時間は短くなっており、情報が出そろうのを待っていると手遅れになります。

判断の基準は次のとおりです。悪用情報を、対応するかどうかの条件ではなく、優先順位を上げる材料として使ってください。

- 悪用が確認されている → 即日対応する
- 認証不要で悪用可能 → 悪用情報がなくても、早めに対応する
- 悪用に高い権限が必要 → 通常の更新サイクルで対応する

## 確認する順番と頻度

情報収集は、頻度を決めて作業を固定してください。判断の手間が消えるため、習慣として続きます。すべてを毎日追う必要はありません。

現実的な運用は、次の2段構えです。重大なものは通知で拾い、それ以外は定期的な確認で処理します。

| タイミング | 確認内容 | 目安時間 |
|---|---|---|
| 随時（通知ベース） | 自分が使っているプラグインに関する重大な情報が出たとき | 15分 |
| 週に1回 | 管理画面の更新一覧を確認し、保留中の更新を適用する | 15分 |
| 月に1回 | 導入プラグインの棚卸しと、設定の診断 | 30分 |

最も重要なのは1行目です。公開から悪用が始まるまでの時間は短く、週次の確認だけでは重大なものに間に合いません。

一方で、すべての脆弱性情報を手作業で追うのは現実的ではありません。導入プラグインが10個あれば照合も10回、これを毎週繰り返す運用は、数週間で必ず止まります。実際、放置されているサイトの多くは、最初から放置されていたわけではなく、途中で運用が形骸化した結果です。

人が判断すべきなのは「対応するかどうか」であって、「脆弱性があるかどうか」を毎回手作業で調べることではありません。ここは仕組みに任せる領域です。

## 情報収集を仕組みにする

手作業を減らす方法は3つあります。無料でできるものから順に挙げます。

いずれも設定は一度きりで、以降は自動的に情報が届きます。導入の手間は数十分ですが、毎週の作業がなくなる効果は大きくなります。

- **更新通知を有効にする**：WordPressやプラグインの更新が公開された際に、メールで通知を受け取る設定です。脆弱性情報そのものではありませんが、修正版が出たことは分かります
- **自動更新を設定する**：マイナー更新やセキュリティ更新だけを自動で適用する運用にすれば、修正版が出た時点で反映されます
- **脆弱性データベースの通知を使う**：導入プラグインと脆弱性情報を自動で照合し、該当したときだけ通知するサービスがあります

自動更新については、業務サイトでは注意が必要です。更新によって表示が崩れる、機能が止まるといった影響が出ることがあります。

## 情報を受け取ったあと、何分で動けるか

情報収集の目的は、知ることではなく、動けることです。脆弱性の公開から悪用が始まるまでの時間は短くなっており、情報を得てから対応方針を考え始めるのでは間に合いません。

対応にかかる時間を短くするには、あらかじめ手順を決めておきます。当日に判断する項目を減らすほど、対応は速くなります。

- **バックアップの取得手順を決めておく**：どのプラグインで、どこへ保存するか。当日に調べていては遅れます
- **停止したときの影響を把握しておく**：そのプラグインを止めるとサイトのどこが動かなくなるかを、平時に確認しておきます
- **連絡先を整理しておく**：制作会社やサーバー会社に確認が必要な場合、連絡先と営業時間を控えておきます

3つ目は軽視されがちですが、実務では大きな差になります。夜間や休日に情報が出た場合、自分で判断できる範囲が決まっていないと、そのまま翌営業日まで放置されます。

## 情報を追う前に、対象の一覧を作る

脆弱性情報の収集は、照合先の一覧がなければ始まりません。導入しているプラグイン・テーマの名前とバージョンを控えた一覧を、先に作ってください。

この一覧は、管理画面から数分で作れます。プラグイン一覧の画面をそのまま記録するだけで構いません。表計算ソフトに転記しておくと、脆弱性情報が出たときに検索で照合できます。

一覧に含めるべき項目は次の4つです。多くを記録しようとすると更新が止まるため、最小限に絞ります。

- **名前**：脆弱性情報に記載される正式名称（日本語の通称ではなく、配布時の名前）
- **現在のバージョン**：影響バージョンとの照合に必須
- **最終更新日**：1年以上前のものは、修正版が出ない可能性を想定しておく
- **止められるかどうか**：脆弱性が公表された際、すぐ停止できる機能か、業務に必須か

4つ目が実務では効きます。修正版がないまま脆弱性が公表されたとき、判断すべきなのは「止めるかどうか」です。この判断を平時に済ませておけば、当日の対応時間が大幅に短くなります。

## 複数のサイトを管理している場合

管理サイトが増えると、手作業での照合は成立しなくなります。3サイトでプラグインが各10個なら、照合対象は30件です。これを毎週確認する運用は現実的ではありません。

複数サイトを扱う場合は、次の順で工数を削ってください。上から順に効果が大きくなります。

1. **プラグインを統一する**：サイトごとに違う製品を使っていると、照合対象がそのまま増えます。同じ機能なら同じ製品に寄せます
2. **プラグインの総数を減らす**：使っていないものを削除します。照合対象が減れば、作業も比例して減ります
3. **通知を仕組み化する**：自分が使っているプラグインに関する情報だけが届く状態を作ります

1番目は見落とされがちですが、効果が大きい対策です。10サイトで同じ5つのプラグインを使っていれば、確認するのは5件で済みます。サイトごとにばらばらの製品を使っていれば、その分だけ確認対象が増えます。

## 現状把握が情報収集の前提になる

ここまで説明した照合作業は、すべて「自分が何をどのバージョンで使っているか」を把握していることが前提です。ここが曖昧なままだと、どれだけ情報を集めても意味がありません。

筆者が開発・公開しているCyberNote Security Checkerは、この前提を整えるための無料の診断プラグインです。WordPress本体・PHP・プラグイン・テーマの更新状況（バージョン鮮度3項目）と、設定のハードニング7項目の計10項目を、管理画面から確認できます。外部への通信を行わず、設定を自動変更しない読み取り専用の設計です。

## よくある質問

**Q. どの情報源から見ればよいですか。**
A. 日本語で確認したいならJVN iPedia、プラグイン単位で調べたいならWPScanが起点になります。

**Q. CVSSの数値が高ければ、必ず優先すべきですか。**
A. 数値だけでは決まりません。認証が不要かどうか、自分が実際にそのプラグインを使っているかを合わせて判断してください。

**Q. 脆弱性情報を毎日確認する必要がありますか。**
A. 必要ありません。重大なものは通知で受け取り、それ以外は週1回の更新確認で対応する運用が現実的です。

**Q. 修正版がまだない脆弱性は、どうすればよいですか。**
A. そのプラグインを一時的に停止するか、代替を探してください。修正版がなければ、更新という手段は使えません。

## まとめ

WordPressの脆弱性情報は、JVN iPedia・CVE／NVD・WPScan・WordPress公式の4つを使い分ければ追えます。ただし、公開される情報の大半は自分のサイトに無関係です。製品名と影響バージョンの2点で機械的に振り分ければ、読むべき情報は大きく減ります。アドバイザリでは、認証の要否・修正版の有無・悪用情報を確認し、CVSSの数値は判断材料の1つとして扱ってください。そして、手作業での照合は必ず形骸化します。人が判断すべきなのは対応するかどうかであり、脆弱性の有無を毎回調べることではありません。

照合の起点となる「何をどのバージョンで使っているか」は、CyberNote Security Checkerで確認できます。管理画面の「プラグイン」→「新規プラグインを追加」で `CyberNote` と検索してください。無料の診断プラグインで、攻撃を遮断する機能はありません。
