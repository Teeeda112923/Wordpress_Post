# WordPressユーザー名漏えい｜author・API

WordPressのユーザー名は、初期状態では外部から取得できます。authorページへのアクセス、REST API、投稿者の表示名。この3つの経路のいずれかで、ログインIDが判明します。ユーザー名が分かれば、攻撃者はパスワードだけを試せばよくなります。この記事では、漏えいの経路を確認する方法と、ログインIDと表示名を分ける設定を示します。

## ユーザー名が漏れると何が起きるか

ユーザー名が外部から取得できると、総当たり攻撃の難易度が大きく下がります。攻撃者はIDとパスワードの両方を推測する必要がなくなり、パスワードだけを試せばよくなるためです。

WordPressのログインには、ユーザー名（ログインID）とパスワードの2つが必要です。このうち片方が既知になれば、攻撃者が試すべき組み合わせは激減します。

さらに問題なのは、ユーザー名の露出が「被害」として認識されにくい点です。露出しているだけでは、サイトは何事もなく動きます。ログにも異常は残りません。攻撃の準備を助けている状態が、静かに続きます。

したがって、この項目は緊急度としては中程度ですが、放置すべきではありません。他の弱点と組み合わさったときに、成功率を押し上げる要素になります。

## ユーザー名が漏れる3つの経路

WordPressでユーザー名が外部に露出する経路は、主に3つです。いずれも初期状態で有効になっており、特別な設定をしなければそのまま公開されています。

経路ごとに、何が見えるかと、確認方法が異なります。自分のサイトで実際に試してみてください。3つとも数分で確認できます。

| 経路 | 何が見えるか | 確認方法 |
|---|---|---|
| authorページ | ユーザー名（スラッグ） | `/?author=1` にアクセス |
| REST API | ユーザー名、表示名、ID | `/wp-json/wp/v2/users` にアクセス |
| 投稿者の表示名 | 表示名（ログインIDと同一の場合、それが露出） | 記事ページの投稿者欄 |

3つ目が見落とされがちです。表示名をログインIDと同じにしていると、記事に表示されている名前がそのままログインIDになります。攻撃者は、URLを叩くまでもなく記事を読むだけで済みます。

以下、経路ごとに詳しく説明します。

### authorページからの露出

ブラウザで `https://（自分のサイト）/?author=1` にアクセスしてください。ユーザー名を含むURL（例：`/author/ユーザー名/`）へ転送される場合、ユーザー名が外部から取得できる状態です。

この `author=1` の数字はユーザーIDで、1は多くの場合、最初に作られた管理者アカウントです。攻撃者は数字を順に変えることで、サイト内のユーザーを列挙できます。これがユーザー列挙と呼ばれる手口です。

### REST APIからの露出

`https://（自分のサイト）/wp-json/wp/v2/users` にアクセスしてください。認証していない状態でも、ユーザーの一覧がJSON形式で返る場合があります。

ここには、ユーザー名（スラッグ）、表示名、IDが含まれます。authorページと違い、一度のアクセスで複数のユーザー情報がまとめて取得できるため、攻撃者にとってはより効率的な経路です。

### 表示名からの露出

WordPressでは、ログインIDとは別に「ブログ上の表示名」を設定できます。ところが、初期状態ではログインIDがそのまま表示名になります。

この状態で記事を公開すると、投稿者欄にログインIDが表示されます。技術的な確認をするまでもなく、誰の目にも見える形で露出することになります。

## ログインIDと表示名を分ける

最初に行うべき対策は、ログインIDと表示名を分けることです。管理画面から数分で設定でき、サイトの動作に影響しません。

手順は次のとおりです。ユーザーごとに設定する必要があるため、投稿者が複数いる場合は全員分を確認してください。

1. 管理画面の「ユーザー」→「プロフィール」を開く
2. 「ニックネーム」に、ログインIDとは異なる名前を入力する
3. 「ブログ上の表示名」で、そのニックネームを選択する
4. 「プロフィールを更新」をクリックする

この設定により、記事の投稿者欄に表示される名前は、ログインIDとは別のものになります。ただし、これだけでは authorページとREST APIからの露出は止まりません。次の対策が必要です。

なお、ニックネームは日本語でも構いません。読者に見せる名前として自然なものを選んでください。

## author slug（ユーザー名スラッグ）を変更する

authorページのURLに含まれる文字列を、ユーザー名スラッグと呼びます。初期状態では、これがログインIDと同一です。

このスラッグをログインIDと別のものに変更すれば、authorページのURLからログインIDは分かりません。標準のWordPressには、スラッグだけを変更する機能がないため、プラグインを使うか、テーマの機能で対応します。

変更する際は、次の点に注意してください。すでに公開しているサイトでは、URLが変わります。

- **既存のauthorページのURLが変わる**：外部からリンクされている場合、リンク切れになります
- **リダイレクトの設定を検討する**：旧URLから新URLへ転送する設定があると安全です
- **すべてのユーザーについて確認する**：投稿者が複数いる場合、全員分の設定が必要です

投稿者アーカイブページを使っていないサイトであれば、そもそもauthorページ自体を無効化する方法もあります。この場合、`/?author=1` へのアクセスは404となり、ユーザー名は露出しません。

## REST APIのユーザー列挙を止める

REST APIからのユーザー情報の取得は、認証していない状態では返さないように制限できます。この設定は、セキュリティプラグインで提供されていることが多くあります。

ここで重要なのは、**REST API全体を無効化しないこと**です。REST APIは、WordPressの管理画面自体や、多くのプラグインが利用しています。全体を止めると、ブロックエディタが動かなくなるなど、サイトの機能に広範な影響が出ます。

止めるべきなのは、ユーザー一覧を返すエンドポイントだけです。プラグインの設定項目に「ユーザー列挙の防止」「REST APIのユーザー情報を非公開にする」といった名称で用意されていることが多いため、その項目を有効にしてください。

設定後、もう一度 `/wp-json/wp/v2/users` にアクセスして確認します。ユーザー情報が返らず、エラーまたは空の応答になっていれば、設定は反映されています。

## コメント欄と外部連携からの露出

authorページ、REST API、表示名の3つを塞いでも、ユーザー名が別の経路から出ていることがあります。見落としやすい2つを挙げます。

1つ目は、コメント欄です。管理者自身がコメントを書き込むと、投稿者名として表示されます。この名前が表示名の設定に従っていない場合、ログインIDが出ることがあります。

2つ目は、外部サービスとの連携です。サイトマップの生成、SNSへの自動投稿、著者情報の構造化データなど、プラグインが出力する情報にユーザー名が含まれることがあります。

確認方法は単純です。サイト内でユーザー名を検索してください。

- ブラウザでページのソースを表示し、ログインIDの文字列を検索する
- サイトマップ（`/sitemap.xml` など）を開き、authorページが含まれていないかを見る
- 構造化データのテスト用ツールで、著者情報に何が出力されているかを確認する

1つ目が最も手軽です。トップページと記事ページのソースを表示し、自分のログインIDが含まれていないかを検索するだけで、大半の露出は見つかります。

これらの経路まで完全に塞ぐのは、現実的には難しい作業です。だからこそ、次章で述べるとおり「漏れても入られない状態」を先に作る必要があります。

## 露出を止めても、それだけでは足りない

ユーザー名の露出を防ぐのは、攻撃者に与える手がかりを減らす対策です。侵入そのものを防ぐ対策ではありません。この位置づけを誤ると、対策の優先順位を間違えます。

考えるべきなのは、「ユーザー名が漏れても侵入されない状態」です。ログインIDが判明しても、次の対策があれば侵入は防げます。

- **十分に長く、使い回していないパスワード**：20文字以上のランダムな文字列にしてください
- **ログイン試行回数の制限**：一定回数の失敗でIPアドレスを一時的に遮断します
- **二要素認証**：パスワードが漏れても、追加の認証がなければ入れません

この3つのほうが、ユーザー名の秘匿より優先度が高い対策です。ユーザー名は、努力しても完全には隠しきれない情報だからです。コメント欄の投稿者名、外部サービスとの連携、テーマの実装。露出の経路は、ここで挙げた3つ以外にも存在しえます。

したがって、順序としてはこうなります。まず、漏れても入られない状態を作る。そのうえで、漏れる経路を減らす。逆にすると、隠すことに労力を使いながら、認証は弱いままという状態になります。

## 露出しているかを確認する

自分のサイトが現在どの状態にあるかは、実際にURLを叩けば分かります。ただし、確認すべき項目はユーザー名の露出だけではありません。

## よくある質問

**Q. ユーザー名が漏れているだけで、被害はありますか。**
A. 直接の被害はありません。ただし、総当たり攻撃の成功率が上がるため、放置すべきではありません。

**Q. 表示名を変えれば、ユーザー名は隠れますか。**
A. 記事の投稿者欄からは隠れますが、authorページとREST APIからは取得できたままです。両方の対策が必要です。

**Q. REST APIを完全に無効化してもよいですか。**
A. 推奨しません。管理画面や多くのプラグインが利用しているため、ブロックエディタが動かなくなるなどの影響が出ます。ユーザー列挙の部分だけを止めてください。

**Q. ユーザー名を隠せば、パスワードは簡単でも大丈夫ですか。**
A. 危険です。ユーザー名は完全には隠しきれません。パスワードの強化と二要素認証のほうが優先度は高くなります。

## まとめ

WordPressのユーザー名は、authorページ、REST API、投稿者の表示名という3つの経路から外部に露出します。いずれも初期状態で有効です。対策としては、まずログインIDと表示名を分け、次にauthor slugを変更し、REST APIのユーザー列挙を止めます。ただし、REST API全体を無効化してはいけません。管理画面やプラグインが利用しているためです。そして、ユーザー名の秘匿は補助的な対策です。優先すべきは、漏れても侵入されない状態、つまり強固なパスワード、ログイン試行回数の制限、二要素認証の3つです。
