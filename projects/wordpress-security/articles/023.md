# CyberNote Security Checker｜診断結果

CyberNote Security Checkerの診断結果は、警告が出た順に直すものではありません。表示される10項目は影響の大きさも作業の難しさも異なるため、順番を誤ると手間だけかかって危険な項目が残ります。この記事では、診断結果の読み方、各項目の判定基準、対応の優先順位、そして修正後の再診断までを、開発者の立場から説明します。

## CyberNote Security Checkerは何を診断するプラグインか

CyberNote Security Checkerは、WordPressサイトの設定とバージョンの状態を10項目で診断し、結果を3段階で表示する無料のプラグインです。攻撃を遮断する機能はなく、現在の状態を可視化することに役割を絞っています。

診断項目は、公式のプラグイン情報（readme）に記載のとおり、2つのカテゴリに分かれます。バージョンが古いことによるリスクと、設定が初期状態のまま残っていることによるリスクは、性質が違うためです。

- **カテゴリA：バージョン鮮度（3項目）** — WordPress本体、PHPバージョン、プラグイン・テーマの更新状況
- **カテゴリB：ハードニング設定（7項目）** — デバッグ表示、ファイル編集機能、管理者ユーザー名、HTTPS、データベース接頭辞、XML-RPC、REST APIによるユーザー名列挙

カテゴリAは「新しくすれば解決する」項目です。一方カテゴリBは、設定を変えることで別の機能に影響が出る可能性があり、判断が必要になります。この違いが、後述する対応順序の考え方につながります。

なお、このプラグインは診断結果を表示するだけで、設定やファイルを自動的に変更することはありません。また、すべての診断はWordPress内部の情報だけを読み取って行われ、外部のサーバーへデータを送信しない設計です。読み取り専用であるため、ほかのセキュリティプラグインと機能が競合することもありません。

## 診断結果の画面をどう読むか

診断結果は、専用の管理画面と、WordPressダッシュボード上のウィジェットの2か所で確認できます。ウィジェットは件数と優先項目の要約、専用画面は10項目すべての内訳を表示します。

まず見るのは、画面上部の集計です。ここに「要対応」「改善推奨」「問題なし」の件数が並びます。この3段階は、その項目が推奨される状態からどれだけ離れているかを示すもので、あなたのサイトにとっての緊急度をそのまま表すものではありません。

3段階の意味は、次のように定義しています。判定そのものと、対応の優先順位は別物である点を、最初に押さえてください。

| 判定 | 意味 | 例 |
|---|---|---|
| 要対応 | 推奨される設定から外れており、放置すると被害の拡大につながりやすい | 本番環境でのデバッグ表示、ファイル編集機能の有効化、HTTPS未対応 |
| 改善推奨 | 直ちに被害が出るとは限らないが、攻撃の足がかりになる | 既定の管理者名、XML-RPCの有効化、REST APIによるユーザー名の露出 |
| 問題なし | 推奨される状態になっている | 対応不要 |

「要対応」が0件でも、安心という意味ではありません。改善推奨の項目は、単体では被害に直結しなくても、組み合わさることでリスクが上がります。たとえば管理者名が admin のまま残っており、同時にユーザー名が外部から取得できる状態であれば、攻撃者はログインIDを推測する必要がなくなります。

逆に、判定が「要対応」でも、すぐに手を出さないほうがよい項目もあります。次章で、項目ごとの中身を確認します。

## 項目別の判定基準と、表示される理由

各項目は、どのような状態のときにどの判定になるかが決まっています。ここでは、判断に迷いやすい項目を中心に説明します。

### バージョン鮮度（カテゴリA）

WordPress本体については、セキュリティ更新が適用されていない状態を検出します。機能追加の更新とセキュリティ更新は緊急度が異なるため、判定でも区別しています。

PHPバージョンは、公式のサポート状況を基準に判定します。サポートが終了したバージョンは「要対応」、セキュリティ修正のみの期間にあるバージョンは「改善推奨」として表示されます。ここで「要対応」と出た場合でも、すぐにサーバーの設定を変えないでください。PHPの更新はプラグインやテーマの動作に影響することがあり、検証なしに実施すると表示が崩れる、機能が止まるといった事態が起こります。

プラグイン・テーマの更新は、保留中の件数と名前を表示します。表示された画面から標準の更新画面へ移動できますが、**プラグイン自身が更新を実行することはありません**。更新するかどうかは、あなたが判断します。

### ハードニング設定（カテゴリB）

デバッグ表示は、エラー内容を画面に出力する設定が本番環境で有効になっている場合に「要対応」となります。ログにのみ記録する設定であれば「改善推奨」です。サーバー内部の情報が画面に表示されることを避けるため、本番環境では出力を止めるのが基本です。

ファイル編集機能は、管理画面からテーマやプラグインのコードを直接書き換えられる状態のときに「要対応」となります。この機能はWordPress公式のハードニング文書でも無効化が推奨されています。管理者アカウントが乗っ取られた場合、この機能を通じてサイト上で任意のプログラムを実行されるためです。

管理者ユーザー名は、admin または administrator という名前のユーザーが存在する場合に「改善推奨」として表示します。「要対応」にしていないのは、名前の変更に移行作業が伴うためです。WordPressはユーザー名の直接変更に対応していないため、新しいアカウントを作って権限と投稿を移し、既存のものを削除する必要があります。

XML-RPCとデータベース接頭辞も、同じ理由で「改善推奨」に留めています。XML-RPCは外部連携で使われていることがあり、無効化すると機能が止まる可能性があります。データベース接頭辞は、稼働中のサイトで変更するとサイトが表示されなくなるリスクがあります。**危険度の高低ではなく、変更に伴うリスクを踏まえた判定**である点を理解してください。

## 診断結果を直す順番

対応は、判定の重い順ではなく、「被害への近さ」と「作業の軽さ」の2軸で決めます。被害に直結し、かつ数分で終わるものから片付けるのが、最も効率のよい進め方です。

診断結果には、すぐ直せるものと、準備が必要なものが混在します。この区別をつけずに上から順に手をつけると、リスクの高い項目が後回しになったまま、作業だけが長引きます。

対応順序の目安は、次のとおりです。この順に処理してください。

| 順番 | 対応する項目 | 理由 |
|---|---|---|
| 1 | デバッグ表示の停止、ファイル編集機能の無効化 | 数分で終わり、侵入後の被害拡大を直接防ぐ |
| 2 | プラグイン・テーマ・WordPress本体の更新 | 公開済みの脆弱性を塞ぐ。バックアップ後に実施 |
| 3 | HTTPSの適用 | 未対応であれば優先度は高いが、証明書の設定が必要 |
| 4 | 管理者ユーザー名、ユーザー名の露出 | 攻撃の前提を崩す。移行作業を伴うため計画的に |
| 5 | XML-RPC | 使用の有無を確認してから判断する |
| 6 | PHPバージョン、データベース接頭辞 | 影響範囲が広く、検証が必要 |

1と2は、当日中に終えられる内容です。ここを片付けるだけで、自動化された攻撃に対する耐性は大きく変わります。

4以降は、作業前に確認すべきことがあります。管理者名の変更では投稿の引き継ぎ設定を誤ると記事が失われますし、XML-RPCの無効化は外部連携を止める可能性があります。バックアップを取得し、1項目ずつ変更してサイトの表示と管理画面を確認しながら進めてください。

## 修正したら再診断する

設定を変更したあとは、再診断を実行して結果が変わったかを確認します。専用画面とダッシュボードウィジェットのどちらにも再診断のボタンがあり、ページを読み込み直さずに最新の状態を取得できます。

再診断で判定が変わらない場合、設定が正しく反映されていない可能性があります。たとえば `wp-config.php` の記述を追加した際、記述する位置が誤っていると設定が有効になりません。ファイルの編集内容と保存先を確認してください。

修正と再診断を1項目ずつ繰り返すことには、もう1つの利点があります。何かの拍子にサイトの表示が崩れたとき、**直前に変更した1項目が原因だと特定できる**ためです。複数の設定をまとめて変更すると、切り分けに時間がかかります。

## 日常の点検につなげる

診断は一度実施して終わりではありません。プラグインを追加したり、サーバーの設定を変えたりすれば、状態は変わります。ダッシュボードのウィジェットは、その変化に気づくために用意しています。

管理画面にログインするたびに、件数が目に入ります。先週まで0件だった「要対応」が増えていれば、その間に加えた変更が何らかのリスクを生んだということです。異常に気づく仕組みを、日常の動線に置いておくのが目的です。

点検の頻度は、次を目安にしてください。作業を固定してしまえば、判断の手間が減り、習慣として続きます。

- **週に1回**：更新の有無を確認する（ウィジェットの件数を見るだけで済みます）
- **月に1回**：専用画面で10項目すべてを確認する
- **プラグインを追加・変更したとき**：その都度、再診断を実行する

なお、無料版で確認できるのは「更新が提供されているかどうか」までです。導入中のプラグインに公開済みの脆弱性が報告されているかを自動で照合する機能は、公式情報のとおり、今後提供予定のPro版で扱う予定です。

このプラグインは診断ツールであり、攻撃を遮断するものではありません。すべての脆弱性を検出できるわけでもないため、防御用のプラグインやバックアップと組み合わせて使ってください。

## よくある質問

**Q. 診断を実行するとサイトに影響がありますか。**
A. ありません。診断は現在の状態を読み取るだけで、設定やファイルを自動的に変更することはありません。

**Q. サイトの情報が外部に送信されますか。**
A. 送信されません。診断はWordPress内部の情報だけを読み取って行われます。

**Q. 「更新画面を開く」を押すと自動で更新されますか。**
A. されません。WordPress標準の更新画面へ移動するだけで、更新するかどうかはあなたが判断します。

**Q. 「要対応」が0件なら安全ですか。**
A. その10項目に大きな不備がなかった、という意味です。未公開の脆弱性や独自に追加したコードの不備は検出できません。

## まとめ

CyberNote Security Checkerの診断結果は、バージョン鮮度3項目とハードニング設定7項目で構成され、要対応・改善推奨・問題なしの3段階で表示されます。判定の重さと対応の優先順位は別物です。デバッグ表示の停止とファイル編集機能の無効化は数分で終わるため最初に、PHPバージョンやデータベース接頭辞は影響範囲が広いため検証のうえ最後に回してください。1項目ずつ直して再診断すれば、問題が起きたときの切り分けも容易になります。
