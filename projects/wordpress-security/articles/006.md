# WordPressセキュリティチェック｜無料診断の手順

WordPressのセキュリティチェックで見るのは、更新状況・設定不備・不要プラグイン・管理者権限の4点です。いずれも無料で確認でき、専用の有料サービスは必要ありません。対策を増やす前に現状を確認すると、すでに塞がっている穴に時間をかけずに済みます。この記事では確認の手順と、見つかった問題を高・中・低に分ける方法を示します。

## WordPressのセキュリティチェックとは何を確認する作業か

WordPressのセキュリティチェックとは、攻撃者が使える入口が開いていないかを、更新状況・設定不備・不要プラグイン・管理者権限の4点から確認する作業です。マルウェアに感染しているかを調べる作業とは目的が異なります。被害が出る前に、入口を見つけて塞ぐための点検です。

なぜこの4点なのか。WordPressへの侵入は、その多くが「古いまま放置されたプログラム」と「初期設定のまま残された項目」を経由して起こるためです。特別な技術で破られるのではなく、開いたままの扉から入られるほうが圧倒的に多いのです。

確認する4点の内容は、次のとおりです。順番にも意味があり、上から順に見ていくと、リスクの大きい順に把握できます。

| 確認する対象 | 何を見るか | 放置した場合のリスク |
|---|---|---|
| 更新状況 | WordPress本体・PHP・プラグイン・テーマのバージョン | 公開済みの脆弱性を突かれる |
| 設定不備 | デバッグ表示、ファイル編集機能、HTTPSなど | 内部情報の露出、侵入後の被害拡大 |
| 不要プラグイン | 使っていないプラグイン・テーマの残存 | 攻撃対象が無駄に増える |
| 管理者権限 | 既定の管理者名、不要アカウント、ユーザー名の露出 | 総当たり攻撃の成功率が上がる |

この4点は、どれか1つを完璧にしても意味がありません。更新を欠かさずにいても、管理者名が admin のままなら、ログインを狙う攻撃の的になります。逆に、認証を固めても古いプラグインを放置していれば、認証を経由せずに侵入される可能性が残ります。

なお、脅威の全体像や被害が出たあとの復旧手順は、別記事で扱っています。本記事は「今の状態を確認する」ところに絞ります。

## 外部診断と管理画面確認は何が違うのか

外部診断は攻撃者から見える情報だけを、管理画面確認はサイト内部の状態だけを示します。守備範囲が重ならないため、両方を行う必要があります。片方だけでは、必ず見落としが出ます。

外部診断とは、サイトのURLを入力するだけで、外部から取得できる情報を調べる方法です。攻撃者と同じ視点に立てるのが利点です。一方、ログインしなければ見えない情報、たとえば停止中のプラグインが残っているかどうかは、外部からは分かりません。

両者の違いを整理すると、次のようになります。どちらが優れているかではなく、見えるものが違うと理解してください。

| 項目 | 外部診断 | 管理画面での確認 |
|---|---|---|
| 確認の起点 | サイトのURL | WordPressへのログイン |
| 分かること | バージョンの露出、ユーザー名の露出、HTTPSの適用状況 | 更新の有無、停止中のプラグイン、アカウント一覧、設定値 |
| 分からないこと | 内部の設定値、停止中のプラグイン、アカウントの数 | 攻撃者から実際に何が見えているか |
| 主な用途 | 攻撃者に与えている手がかりの把握 | 実際に塞ぐべき箇所の特定 |

外部診断で「問題なし」と表示されても、それは外から見た範囲で目立つ問題がなかった、という意味にすぎません。停止中のプラグインが5つ残っていても、外部診断では検出されないことがあります。この限界を知らずに外部診断だけで済ませると、確認したつもりのまま穴が残ります。

逆に、管理画面だけを見ていると、ユーザー名が外部から取得できる状態に気づけません。両方を組み合わせて、初めて現状が見えます。

## 無料でできる確認手順

ここから示す4つの手順は、すべて無料で実施できます。上から順に進めれば、30分程度で現状を把握できます。特別なツールの購入や、外部サービスへの登録は不要です。

作業を始める前に、1つだけ準備をしてください。確認の途中で設定を変更したくなった場合に備え、ファイルとデータベースのバックアップを取得しておきます。確認するだけならサイトに影響はありませんが、修正まで進むなら必要です。

### 更新状況を確認する

WordPress管理画面の「ダッシュボード」→「更新」を開きます。ここに、WordPress本体、プラグイン、テーマの更新可能な項目が一覧で表示されます。更新可能な項目が並んでいる場合、それは公開済みの脆弱性が修正されないまま残っている可能性を示します。

見落とされやすいのが、PHPのバージョンです。PHPとは、WordPressが動作するために必要なプログラム言語のことで、レンタルサーバー側で設定されています。サポートが終了したPHPを使い続けると、脆弱性が見つかっても修正されません。

PHP公式のサポート方針では、各バージョンは2年間の通常サポートと2年間のセキュリティ修正のみの期間を経て、サポートが終了します。PHP 8.1は2025年12月31日にサポートを終了しました。自分のサーバーがどのバージョンで動いているかは、管理画面の「ツール」→「サイトヘルス」→「情報」タブから確認できます。

ただし、PHPの更新はプラグインやテーマの動作に影響することがあります。バックアップを取得し、可能であれば検証環境で確認してから実施してください。

### 設定不備を確認する

設定不備とは、初期状態のまま放置されている項目のうち、攻撃者に有利に働くもののことです。ここでは、影響の大きい3つを挙げます。

1つ目は、デバッグ表示です。デバッグとは、開発中にエラーの内容を画面に表示させる機能で、`wp-config.php` の `WP_DEBUG` という設定で制御されます。本番環境で有効なままだと、サーバー内部のファイルの場所などが画面に表示されることがあります。

2つ目は、管理画面からのファイル編集機能です。WordPressの管理画面には、テーマやプラグインのコードを直接書き換えられる画面があります。管理者アカウントが乗っ取られた場合、この機能を通じてサイト上で任意のプログラムを実行されるため、WordPress公式のハードニング文書でも無効化が推奨されています。

3つ目は、HTTPSの適用です。サイトのURLが `http://` で始まっている場合、ログイン時に入力したパスワードが暗号化されずに送信される可能性があります。

各設定をどう変更するかについては、項目ごとの手順をまとめた別記事があります。本記事では「今どうなっているか」の確認までを扱います。

### 不要プラグイン・テーマを確認する

管理画面の「プラグイン」→「インストール済みプラグイン」を開き、停止中のプラグインがないかを確認します。ここで重要なのは、**停止と削除は違う**という点です。

有効化していなくても、プラグインのファイルはサーバー上に残っています。攻撃者はそのファイルを直接呼び出せる場合があるため、停止しただけでは対策になりません。使っていないものは削除してください。

テーマも同じです。使っていないテーマがいくつも残っているケースは珍しくありません。ただし、テーマは1つだけ予備を残しておくと、有効なテーマに問題が起きたときの切り替え先になります。

削除の前に、そのプラグインが本当に使われていないかを確認してください。フォームや決済など、サイトの表側で動いている機能を止めてしまうと、業務に影響します。判断がつかない場合は、削除せずに一覧を記録しておき、制作会社や開発者に確認してから進めます。

### 管理者権限を確認する

管理画面の「ユーザー」を開き、権限グループが「管理者」になっているアカウントを確認します。確認するのは次の3点です。

- **既定の管理者名が残っていないか**：ユーザー名が admin や administrator の場合、攻撃者が最初に試す名前と一致します
- **不要なアカウントが残っていないか**：退職した担当者や、制作会社のアカウントが管理者のまま残っているケースがあります
- **ユーザー名が外部に露出していないか**：ブラウザで `https://（自分のサイト）/wp-json/wp/v2/users` を開き、ユーザー情報が表示されるかを確認します

3つ目は、外部診断にあたる確認です。ここでユーザー名が表示される場合、攻撃者はログインIDを推測する必要がなくなり、パスワードだけを試せばよくなります。総当たり攻撃の難易度が半分になる、と考えてください。

ただし、管理者名の変更には注意が必要です。WordPressはユーザー名の直接変更に対応していないため、新しいアカウントを作って権限と投稿を移し、既存のアカウントを削除するという手順を踏みます。投稿の引き継ぎ設定を誤ると記事が失われるため、必ずバックアップを取ってから作業してください。

## プラグインで10項目を一括診断する

ここまでの確認を手作業で行うと、項目が多く、確認漏れが起こります。診断プラグインを使えば、同じ内容を管理画面から一度に確認できます。毎月続ける前提であれば、手作業より現実的です。

WordPressには標準で「サイトヘルス」機能があり、バージョンやサーバー環境の一部を確認できます。ただし、既定の管理者名の有無やユーザー名の露出といった項目は含まれていません。ここを補うのが診断プラグインの役割です。

筆者が開発・公開しているCyberNote Security Checkerは、この4つの観点を10項目に分けて診断する無料のプラグインです。公式のプラグイン情報（readme）に記載のとおり、診断はバージョン鮮度に関する3項目と、ハードニング設定に関する7項目で構成されています。

- **カテゴリA：バージョン鮮度（3項目）** — WordPress本体、PHPバージョン、プラグイン・テーマの更新状況
- **カテゴリB：ハードニング設定（7項目）** — デバッグ表示、ファイル編集機能、管理者ユーザー名、HTTPS、データベース接頭辞、XML-RPC、REST APIによるユーザー名列挙

結果は「問題なし」「改善推奨」「要対応」の3段階で表示され、各項目に理由と対処の説明が付きます。外部への通信を行わず、設定を自動で変更しない読み取り専用の設計であるため、既存の環境やほかのプラグインに影響を与えません。

導入は、管理画面の「プラグイン」→「新規プラグインを追加」で `CyberNote` と検索し、インストールして有効化するだけです。なお、これは状態を確認するための診断ツールであり、攻撃を遮断するツールではありません。すべての脆弱性を検出できるわけでもないため、更新とバックアップは引き続き必要です。

## 検出された問題を高・中・低に分ける

チェックを行うと、たいてい複数の問題が同時に見つかります。すべてを一度に直そうとすると途中で止まるため、「被害への近さ」と「作業難度」の2軸で優先度を分けてください。被害に直結し、かつ作業が軽いものから処理します。

ここで注意したいのは、診断ツールが示す3段階の判定と、対応の優先度は別物だという点です。判定は「その項目の状態」を表すもので、あなたのサイトにとっての緊急度をそのまま示すわけではありません。たとえばデータベース接頭辞の変更は、稼働中のサイトでは作業リスクが高く、後回しにするのが妥当な場合があります。

優先度の判断基準は、次のとおりです。この基準で並べ替えてから着手してください。

| 優先度 | 判断基準 | 該当する例 |
|---|---|---|
| 高 | 侵入や情報漏えいに直結し、作業も軽い | 更新可能なプラグイン・本体の適用、デバッグ表示の停止、ファイル編集機能の無効化、HTTPS未対応の解消 |
| 中 | 攻撃の足がかりになるが、単体では被害に直結しない | 既定の管理者名、ユーザー名の露出、不要プラグインの削除、XML-RPCの扱い |
| 低 | 効果はあるが、作業リスクや影響範囲が大きい | データベース接頭辞の変更、サポート終了PHPの更新（検証が必要） |

低に分類した項目は「やらなくてよい」という意味ではありません。作業に検証や準備が必要なため、時間を確保して計画的に行う、という区分です。とくにPHPの更新は影響範囲が広いため、バックアップと動作確認を前提にしてください。

高の項目は、多くが数分で終わります。まずここを片付けるだけで、自動化された攻撃に対する耐性は大きく変わります。

## チェック後にやってはいけないこと

診断結果を見た直後に、表示された項目をまとめて修正する。これが最も事故を招きます。検出された項目のなかには、環境によっては変更してはいけないものが含まれるためです。

代表的なのがXML-RPCです。これは外部アプリケーションからWordPressを操作するための仕組みで、無効化すればログイン試行の経路を1つ減らせます。しかし、Jetpackや一部のモバイルアプリ、外部連携ツールがこの仕組みを使っています。使っている場合に無効化すると、その機能が動かなくなります。

同様に、ログイン画面のURL変更や管理画面のIP制限は、設定を誤ると自分自身がログインできなくなります。復旧にはFTPやサーバーのファイルマネージャーでの作業が必要になるため、その手順を先に確認してから設定してください。

作業前に守るべき点をまとめます。急いで直すより、戻せる状態を作ってから進めるほうが、結果的に早く終わります。

- 変更前に、ファイルとデータベースのバックアップを取得する
- 一度に複数の項目を変更せず、1つ変更したらサイトの表示と管理画面を確認する
- XML-RPC、データベース接頭辞、PHPバージョンは、影響範囲を確認してから判断する
- ログイン系の設定は、締め出された場合の復旧手順を先に把握しておく

この4点を守れば、確認から修正までを安全に進められます。診断はあくまで現状を知るための作業であり、そこから先の判断は、サイトの使われ方を知っている運営者にしかできません。

## よくある質問

**Q. WordPressのセキュリティチェックは無料でできますか。**
A. できます。管理画面での確認、ブラウザからの外部確認、診断プラグインの利用は、いずれも無料で実施できます。

**Q. どのくらいの頻度で行えばよいですか。**
A. 更新状況の確認は週に1回、4項目全体の確認は月に1回が目安です。プラグインを追加したあとにも確認してください。

**Q. セキュリティプラグインを入れていれば、チェックは不要ですか。**
A. 不要にはなりません。攻撃を遮断するプラグインは、既定の管理者名やユーザー名の露出といった設定の問題を解消するものではありません。

**Q. 診断で「問題なし」と表示されたら安全ですか。**
A. その診断で確認した範囲に問題が見つからなかった、という意味です。未公開の脆弱性や独自に追加したコードの不備は検出できないため、バックアップと更新は続けてください。

## まとめ

WordPressのセキュリティチェックは、更新状況・設定不備・不要プラグイン・管理者権限の4点を確認する作業です。外部診断は攻撃者から見える情報を、管理画面の確認はサイト内部の状態を示すため、両方が必要になります。検出された問題は、被害への近さと作業難度の2軸で高・中・低に分け、高から順に処理してください。更新の適用やデバッグ表示の停止といった「高」の項目は、多くが数分で終わります。
