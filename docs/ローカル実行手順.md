# ローカル実行手順（CORESERVER / Imunify360 対策）

GitHub Actions のランナーは**米国データセンターのIP**のため、CORESERVER の
Imunify360 ボット保護に「自動化アクセス」と判定されて遮断されます
（ログの `Access denied by Imunify360 bot-protection` がこれ）。

そこで、**自分のPC（日本の通常回線＝固定的なIP）から直接実行**します。
これで海外データセンター判定・ボット判定を回避できます。

---

---

## ★ Windows（コマンドプロンプト）の方へ

Mac/Linux 用コマンド（`source`・`cp`・`MODE=x ./run_local.sh` など）は
Windows では動きません。Windows では以下を使ってください。

### 準備（初回のみ）
```bat
git clone https://github.com/Teeeda112923/Wordpress_Post.git
cd Wordpress_Post
python -m pip install -r requirements.txt
copy .env.example .env
notepad .env
```
`notepad .env` で開いたら、3行を実際の値にして保存：
```
WP_BASE_URL=https://cybernote.click
WP_USERNAME=あなたのWPユーザー名
WP_APP_PASSWORD=xxxx xxxx xxxx xxxx xxxx xxxx
```
※ venv（仮想環境）は必須ではありません。上記のように直接 `pip install` すればそのまま動きます。

### 実行（`run_local.bat` を使う）
```bat
rem まず疎通確認（投稿しない dry-run）
set NOS=1,2
run_local.bat

rem 問題なければ 2件を下書き投稿（既存があれば更新）
set MODE=post
set STATUS=draft
set WRITE_MODE=upsert
set NOS=1,2
run_local.bat

rem 全50件を公開
set MODE=post
set STATUS=publish
set WRITE_MODE=upsert
set NOS=
run_local.bat
```
- `set` した値はそのウィンドウで残ります。**dry-run に戻したいときは `set MODE=dry-run`**、
  対象を全件に戻すときは `set NOS=` のように空で上書きしてください（cmd を開き直してもOK）。
- 実行直後に出る `MODE=... STATUS=...` の行で、今の設定を必ず確認できます。

### 直接コマンドで実行したい場合（バッチを使わない）
1行で貼り付けても動きます（dry-run で No.1,2 を確認する例）：
```bat
python wp_auto_post.py --input data/wordpress-security-production.xlsx --articles-dir projects/wordpress-security/articles --images-dir projects/wordpress-security/eyecatches --post-status draft --write-mode create_only --nos 1,2 --output-dir projects/wordpress-security/results --dry-run
```

> 以下（第1〜6章）は Mac/Linux 向けの説明です。Windows の方は上の内容だけで完結します。

## 1. 一度だけの準備

### 1-1. リポジトリを手元に用意
```bash
git clone https://github.com/Teeeda112923/Wordpress_Post.git
cd Wordpress_Post
```
（すでにある場合は `git pull` で最新にしてください）

### 1-2. Python 環境を作る（Python 3.10 以上）
```bash
python3 -m venv .venv
source .venv/bin/activate          # Windowsは .venv\Scripts\activate
pip install -r requirements.txt
```

### 1-3. 接続情報を .env に書く
```bash
cp .env.example .env
```
`.env` を開いて、実際の値を入れます（`.env` は Git に上がりません）:
```
WP_BASE_URL=https://cybernote.click
WP_USERNAME=あなたのWPユーザー名
WP_APP_PASSWORD=xxxx xxxx xxxx xxxx xxxx xxxx
```
- `WP_APP_PASSWORD` は WordPress 管理画面 →「ユーザー」→ プロフィール →
  「アプリケーションパスワード」で発行したもの（スペースはそのままでOK）。

---

## 2. 実行（`run_local.sh` を使う）

> 毎回 `source .venv/bin/activate` してから実行してください。

### 手順A：まず疎通確認（投稿しない dry-run）
```bash
MODE=dry-run NOS=1,2 ./run_local.sh
```
- ログに `認証OK: 〇〇 としてログイン`（ユーザー名が出る）＝ **ブロック突破OK**。
- `Access denied by Imunify360` が**出なければ成功**です。

### 手順B：少数で本番テスト（2件を下書き投稿）
```bash
MODE=post STATUS=draft NOS=1,2 ./run_local.sh
```
- 既存の下書き2件がある場合は、更新したいなら `WRITE_MODE=upsert` を付けます:
  ```bash
  MODE=post STATUS=draft WRITE_MODE=upsert NOS=1,2 ./run_local.sh
  ```
- WordPress 管理画面で、本文・アイキャッチ・Rank Math のフォーカスキーワードを確認。

### 手順C：全50件を投稿
```bash
# まず全件を下書きで作る場合
MODE=post STATUS=draft ./run_local.sh

# いきなり公開する場合（既存は更新、無ければ新規＝重複しない）
MODE=post STATUS=publish WRITE_MODE=upsert ./run_local.sh
```

---

## 3. オプション早見表

`run_local.sh` の前に付ける環境変数で挙動を変えられます。

| 変数 | 既定値 | 説明 |
|------|--------|------|
| `MODE` | `dry-run` | `dry-run`＝確認のみ / `post`＝実際に投稿 |
| `STATUS` | `draft` | `draft`（下書き）/ `pending`（レビュー待ち）/ `publish`（公開） |
| `WRITE_MODE` | `create_only` | `create_only`＝新規のみ / `update_only`＝更新のみ / `upsert`＝両方 |
| `NOS` | （空＝全件） | 対象No。例 `1,2` や `3-10,15` |
| `LIMIT` | `0` | 件数上限。`0` で全件 |
| `CATEGORY` | （空） | 全記事のカテゴリを統一。空ならExcelのWPカテゴリ列 |

例：
```bash
MODE=post STATUS=publish WRITE_MODE=upsert NOS=1-10 LIMIT=0 ./run_local.sh
```

---

## 4. 重複を出さないための考え方

- 既存記事の有無は **スラッグ → タイトル完全一致** の順で判定します
  （下書きはスラッグが空になりがちなので、タイトルでも探します）。
- `create_only`：既存があれば**スキップ**（新規だけ作る）
- `update_only`：既存があれば**更新**、無ければスキップ
- `upsert`：既存は**更新**、無ければ**新規**（最も安全に「あるべき状態」に揃う）
- **ローカル実行ならブロックされないので、この既存判定が正しく効きます。**

---

## 5. うまくいかないときの切り分け

| 症状 | 見るところ |
|------|-----------|
| `認証OK: (unknown)` / `Access denied by Imunify360` | まだブロックされている。回線を変える／CORESERVER側でそのIPを許可 |
| `WP_BASE_URL が未設定` 等 | `.env` の3項目が入っているか確認 |
| `401 / 403` 認証エラー | アプリケーションパスワードを再発行して `.env` を更新 |
| `term_exists 400` | カテゴリ/タグが既存。ブロックが解ければ既存判定で回避されます |

---

## 6. GitHub Actions は使わないの？

Actions は便利ですが、IPが米国データセンターのため CORESERVER の Imunify360 に
弾かれます。**投稿の実行だけはローカル（日本の回線）から**行うのが安定します。
記事や画像の生成・管理は今まで通り GitHub 上で運用してかまいません。
